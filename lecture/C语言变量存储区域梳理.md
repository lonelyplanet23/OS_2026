# C 语言变量存储区域梳理

> 主题：一个 C 程序中，不同类型、不同作用域、不同存储期的变量分别存放在进程地址空间的哪个区域。

## 1. 先给结论：变量到底放在哪？

判断一个变量放在哪里，不能只看“是不是 const”或“是不是局部变量”，更关键的是看：

1. **存储期**：它活多久？
   - 静态存储期：程序整个运行期间都存在。
   - 自动存储期：进入代码块创建，离开代码块销毁。
   - 动态存储期：由 `malloc` / `free` 控制。
   - 线程存储期：每个线程一份，线程存在期间有效。
2. **是否初始化**：静态存储期对象是显式非零初始化，还是未初始化/零初始化？
3. **是否只读**：是否能放进只读数据区。
4. **变量本身与它指向的对象要分开看**：指针变量放在哪里，和指针指向的内存放在哪里，是两个问题。

### 1.1 总览表

| C 语言写法/对象 | 典型存储区域 | 生命周期 | 作用域 | 关键说明 |
|---|---|---|---|---|
| 函数代码 `void f(){}` | `.text` / 代码段 | 程序运行期间 | 按声明决定 | 存放机器指令，通常只读、可执行 |
| 字符串字面量 `"hello"` | `.rodata` / 只读数据段 | 程序运行期间 | 表达式中可用 | 内容通常只读，不应修改 |
| 已初始化全局变量 `int g = 1;` | `.data` | 程序运行期间 | 全局/文件作用域 | 有初值且需要占用可执行文件中的初始化数据 |
| 未初始化全局变量 `int g;` | `.bss` | 程序运行期间 | 全局/文件作用域 | 程序装载时清零，通常不占可执行文件实际数据内容 |
| 零初始化全局变量 `int g = 0;` | `.bss` 通常如此 | 程序运行期间 | 全局/文件作用域 | 显式写 0 也常被放入 `.bss` |
| 已初始化静态全局变量 `static int g = 1;` | `.data` | 程序运行期间 | 当前源文件 | `static` 改变链接属性，不改变“静态存储期”本质 |
| 未初始化静态全局变量 `static int g;` | `.bss` | 程序运行期间 | 当前源文件 | 文件内可见，存储仍在静态数据区 |
| 已初始化静态局部变量 `static int x = 1;` | `.data` | 程序运行期间 | 块作用域 | 作用域像局部变量，生命周期像全局变量 |
| 未初始化静态局部变量 `static int x;` | `.bss` | 程序运行期间 | 块作用域 | 只初始化一次，默认清零 |
| 普通局部变量 `int x;` | 栈 `stack`，也可能被优化到寄存器 | 块执行期间 | 块作用域 | 自动变量，函数返回后失效 |
| 局部数组 `int a[100];` | 栈 `stack` | 块执行期间 | 块作用域 | 数组较大时容易导致栈溢出 |
| 函数参数 `void f(int x)` | 栈或寄存器 | 调用期间 | 函数体内 | 由调用约定决定，逻辑上属于当前调用栈帧 |
| `malloc` 得到的对象 | 堆 `heap` | `malloc` 到 `free` | 由指针可达性决定 | 动态分配，必须正确释放 |
| 指针变量 `int *p` | 看 `p` 自身的声明位置 | 看 `p` 自身 | 看 `p` 自身 | `p` 在哪里和 `*p` 指向哪里无必然关系 |
| 全局 `const int c = 1;` | 通常 `.rodata`，也可能 `.data` | 程序运行期间 | 全局/文件作用域 | 若确实只读，编译器倾向放只读段 |
| 局部 `const int c = 1;` | 栈或寄存器 | 块执行期间 | 块作用域 | `const` 不等于一定进 `.rodata` |
| `extern int x;` | 不分配新空间 | 取决于定义处 | 当前声明可见 | `extern` 声明不是定义，真正定义决定位置 |
| `_Thread_local int x;` | TLS 线程局部存储区 | 当前线程期间 | 按声明决定 | 每个线程各有一份副本 |

## 2. 进程地址空间：这些“段”在运行时是什么样？

课程资料中强调：程序装入内存后，不只是可执行文件中的内容，还会构建运行时所需的栈和堆。一个典型进程虚拟地址空间可以抽象成：

```text
高地址
+-----------------------------+
| stack 栈                     |
| - 普通局部变量               |
| - 函数参数/返回地址/调用现场  |
| - 每个线程通常有自己的栈      |
+-----------------------------+
| mmap / 共享库映射区           |
| - 动态链接库                 |
| - 文件映射                   |
+-----------------------------+
| heap 堆                      |
| - malloc/calloc/realloc       |
| - 程序员 free 释放            |
+-----------------------------+
| .bss 未初始化/零初始化数据    |
| - int g;                     |
| - static int x;              |
+-----------------------------+
| .data 已初始化可写数据        |
| - int g = 1;                 |
| - static int x = 1;          |
+-----------------------------+
| .rodata 只读数据              |
| - 字符串字面量               |
| - 只读常量                   |
+-----------------------------+
| .text 代码段                  |
| - 机器指令                   |
+-----------------------------+
低地址
```

这只是经典示意图。真实系统会受到 ABI、编译器、链接器、操作系统、地址空间布局随机化 ASLR、共享库加载方式等影响，但核心分类不变。

## 3. ELF 文件里的 Section 与运行时 Segment

课程资料中提到 ELF 文件视图和执行视图要区分：

- **Section**：链接视角，更关心文件内部如何组织。
- **Segment**：装载视角，更关心操作系统加载器如何把内容映射进进程地址空间。

常见 Section：

| Section | 内容 |
|---|---|
| `.text` | 可执行机器代码 |
| `.rodata` | 只读数据，如字符串常量、只读常量 |
| `.data` | 已初始化的全局变量、静态变量 |
| `.bss` | 未初始化或零初始化的全局变量、静态变量，只预留空间 |
| `.symtab` | 符号表，供链接/调试使用 |
| `.rela.text` | 重定位信息，供链接使用 |

运行时 OS 加载器通常按 Segment 映射，例如：

- 只读可执行 LOAD Segment：可能包含 `.text`、部分 `.rodata`。
- 可读可写 LOAD Segment：可能包含 `.data`、`.bss`。

重点：

1. `.text`、`.rodata`、`.data` 通常在可执行文件中有实际内容。
2. `.bss` 通常在可执行文件中只记录大小，不保存一大堆 0。
3. 程序装载时，加载器为 `.bss` 分配内存并清零。

这解释了为什么：

```c
int a[1000000];
```

如果它是全局数组，通常进 `.bss`，可执行文件不会因为数组全是 0 就真的增大 4MB；而如果写成：

```c
int a[1000000] = {1};
```

它有非零初始化内容，通常会进入 `.data`，可执行文件可能显著变大。

## 4. 全局变量：看初始化情况

### 4.1 已初始化全局变量进 `.data`

```c
int g1 = 10;
char g2 = 'A';
```

这些变量具有：

- 文件作用域或外部链接属性。
- 静态存储期。
- 程序开始运行前完成初始化。
- 程序结束时才消失。

因为它们有明确的非零初值，加载器需要从可执行文件中读取初始化内容，所以通常放在 `.data`。

### 4.2 未初始化/零初始化全局变量进 `.bss`

```c
int g3;
int g4 = 0;
static int g5;
static int g6 = 0;
```

这些通常放在 `.bss`。

`.bss` 的核心特征：

- 属于静态内存分配。
- 程序运行期间一直存在。
- 程序装载时由系统清零。
- 可执行文件中通常只记录大小，不存储实际 0 数据。

所以：

```c
int g;
```

不是“随机值”。作为全局变量，它有静态存储期，会被默认初始化为 0。

## 5. `static` 变量：作用域变小，生命周期不变短

`static` 是最容易混淆的关键字之一。它在不同位置含义不同，但和变量存储区域的关系可以这样记：

> 只要一个对象具有静态存储期，它通常就不在普通函数调用栈上，而是在静态数据区：`.data` 或 `.bss`。

### 5.1 文件作用域 `static` 全局变量

```c
static int file_x = 1;
static int file_y;
```

- `file_x`：已初始化，进 `.data`。
- `file_y`：未初始化，进 `.bss`。

这里的 `static` 主要影响**链接属性**：变量只在当前源文件内可见，不能被其他源文件直接 `extern` 引用。

### 5.2 函数内部 `static` 局部变量

```c
void f(void) {
    static int count = 0;
    count++;
}
```

`count` 的作用域在函数块内部，但它不是普通栈变量。它：

- 只初始化一次。
- 程序整个运行期间都存在。
- 函数返回后值仍然保留。
- 因为 `= 0`，通常放在 `.bss`；如果是 `= 1`，通常放在 `.data`。

对比：

```c
void f(void) {
    int a = 0;
    static int b = 0;
    a++;
    b++;
}
```

- `a` 每次调用重新创建，通常在栈上。
- `b` 只存在一份，跨调用保留，通常在 `.bss`。

## 6. 局部变量：普通局部变量通常在栈上

```c
void f(int n) {
    int x = 1;
    int arr[100];
}
```

`x` 和 `arr` 是自动存储期对象，通常存放在当前函数调用的栈帧中。

栈中通常包括：

- 普通局部变量。
- 函数参数。
- 返回地址。
- 保存的寄存器。
- 调用现场信息。

函数调用时创建栈帧，函数返回时销毁栈帧。因此：

```c
int *bad(void) {
    int x = 10;
    return &x;
}
```

这是错误的，因为 `x` 是栈上局部变量，函数返回后生命周期结束，返回它的地址会形成悬空指针。

### 6.1 局部变量不一定真的“落在内存栈上”

编译器优化后，普通局部变量可能：

- 放在寄存器里。
- 被常量传播优化掉。
- 被内联后不再有独立存储位置。

但从语言语义和操作系统内存模型角度，可以理解为自动存储期对象属于“栈帧管理”。考试或课程题目中通常回答“普通局部变量在栈上”。

## 7. 堆：`malloc` 分配的是对象，不是指针变量本身

```c
void f(void) {
    int *p = malloc(sizeof(int));
    *p = 123;
    free(p);
}
```

这里有两个东西：

| 对象 | 存储位置 |
|---|---|
| 指针变量 `p` 本身 | 普通局部变量，通常在栈上 |
| `malloc` 返回的那块 `sizeof(int)` 内存 | 堆上 |

堆的特点：

- 运行时动态分配。
- 生命周期由程序员控制。
- `malloc` / `calloc` / `realloc` 分配。
- `free` 释放。
- 函数返回不会自动释放堆对象。

例如：

```c
int *ok(void) {
    int *p = malloc(sizeof(int));
    *p = 10;
    return p;
}
```

这比返回局部变量地址合理，因为 `malloc` 的对象在堆上，函数返回后仍然存在。但调用者之后必须 `free`。

常见错误：

```c
void leak(void) {
    int *p = malloc(100);
}
```

函数返回后，`p` 这个栈上指针变量消失了，但堆上的 100 字节没有释放，也没有指针能找到它，造成内存泄漏。

## 8. `const`：不是判断存储区域的唯一标准

很多人会误以为“`const` 一定在常量区”。这不准确。

### 8.1 字符串字面量通常在 `.rodata`

```c
char *p = "hello";
```

- `p` 是指针变量。如果它是局部变量，`p` 本身通常在栈上。
- `"hello"` 是字符串字面量，通常在 `.rodata`。
- 修改字符串字面量是未定义行为。

所以：

```c
p[0] = 'H';
```

不合法，可能导致程序崩溃。

更安全的写法：

```c
char s[] = "hello";
s[0] = 'H';
```

这里：

- `"hello"` 仍可作为初始化数据来源。
- `s` 是局部数组，数组对象通常在栈上。
- 数组内容是字符串的一份可修改副本。

### 8.2 全局 `const` 通常可放 `.rodata`

```c
const int c = 10;
```

如果是全局对象，且编译器/链接器确认它只读，通常放在 `.rodata`。

但是否一定放 `.rodata` 受平台、链接属性、编译器实现影响。考试中通常答“全局 const 常量在只读数据段”。

### 8.3 局部 `const` 通常仍在栈或寄存器

```c
void f(void) {
    const int x = 10;
}
```

`x` 是局部自动变量，只是不能通过 `x` 这个名字修改它。它通常仍属于当前栈帧，或者被优化成寄存器/立即数。

所以要记住：

> `const` 限制的是“能不能通过这个左值修改”，不直接决定“对象一定放在哪个段”。

## 9. 指针变量：一定要区分“指针本身”和“指向内容”

这是最常考、也最容易错的点。

```c
char *p1 = "abc";
char p2[] = "abc";
static char *p3 = "abc";
```

逐个分析：

### 9.1 `char *p1 = "abc";`

如果在函数内部：

- `p1` 本身：局部指针变量，通常在栈上。
- `"abc"`：字符串字面量，通常在 `.rodata`。
- `p1` 的值：保存 `.rodata` 中字符串首地址。

### 9.2 `char p2[] = "abc";`

如果在函数内部：

- `p2` 是局部数组，数组整体通常在栈上。
- 内容是 `'a' 'b' 'c' '\0'` 的可修改副本。

### 9.3 `static char *p3 = "abc";`

如果在函数内部：

- `p3` 本身是静态局部指针变量，已初始化，通常在 `.data`。
- `"abc"` 仍然通常在 `.rodata`。
- `p3` 保存字符串字面量地址。

再看一个经典例子：

```c
int g = 1;

void f(void) {
    int x = 2;
    int *p = &g;
    int *q = &x;
}
```

| 名字 | 本身在哪里 | 指向哪里 |
|---|---|---|
| `g` | `.data` | 不是指针 |
| `x` | 栈 | 不是指针 |
| `p` | 栈 | 指向 `.data` 中的 `g` |
| `q` | 栈 | 指向栈中的 `x` |

## 10. `extern`：声明不分配空间，定义才分配空间

```c
extern int x;
```

这只是告诉编译器：“别的地方有一个 `int x`”。它通常不分配存储空间。

真正分配空间的是定义：

```c
int x = 1;   // 定义，通常 .data
int y;       // 定义，通常 .bss
```

如果多个源文件中写：

```c
extern int x;
```

它们只是引用同一个外部定义。`x` 到底在 `.data` 还是 `.bss`，取决于那个真正的定义有没有非零初始化。

## 11. 函数、函数指针与代码段

```c
void foo(void) {}

void f(void) {
    void (*fp)(void) = foo;
}
```

- `foo` 的机器指令在 `.text`。
- `fp` 是局部函数指针变量，通常在栈上。
- `fp` 的值是函数 `foo` 在代码段中的地址。

函数名在表达式中常转换为函数地址，但函数本体不是普通数据对象，它的机器代码位于代码段。

## 12. 线程：共享什么，私有什么？

课程资料中提到，线程共享进程的地址空间和打开文件等资源，但线程有自己的寄存器和栈。

在多线程程序中：

| 区域/资源 | 多线程之间是否共享 |
|---|---|
| `.text` 代码段 | 共享 |
| `.rodata` 只读数据段 | 共享 |
| `.data` 全局/静态可写数据 | 共享 |
| `.bss` 全局/静态零初始化数据 | 共享 |
| heap 堆 | 共享 |
| 打开的文件 | 通常共享 |
| 栈 stack | 每个线程独立一份 |
| 寄存器/程序计数器 | 每个线程独立一份 |
| TLS 线程局部存储 | 每个线程独立副本 |

例如：

```c
int global_count;

void *thread_func(void *arg) {
    int local = 0;
    static int shared_static = 0;
    _Thread_local static int tls_value = 0;
}
```

- `global_count`：`.bss`，所有线程共享。
- `local`：当前线程栈上，每个线程调用时各有自己的局部变量。
- `shared_static`：`.bss` 或 `.data`，所有线程共享同一个对象。
- `tls_value`：TLS，每个线程一份。

所以，多线程中全局变量、静态变量、堆对象需要考虑同步；普通局部变量通常天然线程私有。

## 13. 虚拟内存视角：这些地址都是虚拟地址

课程资料中强调，虚拟内存为每个进程提供一个大的、一致的、连续可用的、私有地址空间。程序看到的 `.text`、`.data`、堆、栈地址，本质上都是**虚拟地址**。

硬件和操作系统通过：

- MMU
- 页表
- TLB
- 缺页中断
- 页面置换

把虚拟地址转换成物理地址，并按需把页面调入内存。

所以：

```c
printf("%p\n", &g);
printf("%p\n", &x);
```

打印出来的是虚拟地址，不是物理内存地址。

这也解释了为什么不同进程可以看到“相似甚至相同”的虚拟地址，但它们不会互相覆盖：每个进程有自己的页表，虚拟地址到物理地址的映射不同。

## 14. fork、exec 与地址空间

虽然这个问题核心是变量在哪个段，但理解进程模型时也要知道：

### 14.1 `fork`

`fork` 创建子进程，子进程获得父进程地址空间的逻辑副本。

- `.text`、`.rodata` 可共享物理页。
- `.data`、`.bss`、heap、stack 逻辑上复制。
- 现代系统通常用写时复制 Copy-on-Write，只有写入时才真正复制物理页。

### 14.2 `exec`

`exec` 用新程序替换当前进程的地址空间。

- 原来的代码段、数据段、堆栈等被新程序映像替换。
- 新程序重新建立自己的 `.text`、`.data`、`.bss`、heap、stack 等。

## 15. 用一段完整 C 代码逐行判断

```c
#include <stdlib.h>

int g1 = 10;
int g2;
static int g3 = 20;
static int g4;
const int g5 = 30;
char *g6 = "hello";
char g7[] = "world";

void func(int param) {
    int a = 1;
    const int b = 2;
    static int c = 3;
    static int d;
    char *p = "abc";
    char arr[] = "def";
    int *heap = malloc(sizeof(int));
    *heap = 100;
    free(heap);
}
```

逐项分析：

| 名字/对象 | 存储区域 | 理由 |
|---|---|---|
| `g1` | `.data` | 全局变量，非零初始化 |
| `g2` | `.bss` | 全局变量，未初始化，默认清零 |
| `g3` | `.data` | 静态全局变量，非零初始化 |
| `g4` | `.bss` | 静态全局变量，未初始化 |
| `g5` | 通常 `.rodata` | 全局 const，只读对象 |
| `g6` 本身 | `.data` | 全局指针变量，初始化为字符串地址 |
| `"hello"` | `.rodata` | 字符串字面量 |
| `g7` | `.data` | 全局字符数组，有初始化内容，数组本身可写 |
| `func` 机器指令 | `.text` | 函数代码 |
| `param` | 栈或寄存器 | 函数参数，调用期间存在 |
| `a` | 栈或寄存器 | 普通局部变量 |
| `b` | 栈或寄存器 | 局部 const 自动变量 |
| `c` | `.data` | 静态局部变量，非零初始化 |
| `d` | `.bss` | 静态局部变量，未初始化 |
| `p` 本身 | 栈 | 局部指针变量 |
| `"abc"` | `.rodata` | 字符串字面量 |
| `arr` | 栈 | 局部字符数组，是可修改副本 |
| `"def"` | 可能在 `.rodata` 或作为初始化数据处理 | 用于初始化局部数组，最终 `arr` 在栈上 |
| `heap` 本身 | 栈 | 局部指针变量 |
| `malloc(sizeof(int))` 返回对象 | 堆 | 动态分配内存 |

特别注意：

```c
char *g6 = "hello";
char g7[] = "world";
```

二者完全不同：

- `g6` 是一个指针，指向只读字符串。
- `g7` 是一个数组，数组对象本身在 `.data`，内容可修改。

## 16. 高频易错点

### 易错点 1：`static` 局部变量不是栈变量

```c
void f(void) {
    static int x;
}
```

`x` 在 `.bss`，不是栈。它只是在语法作用域上只能在函数内访问。

### 易错点 2：局部 `const` 不一定在只读数据段

```c
void f(void) {
    const int x = 1;
}
```

`x` 通常是栈上自动变量，甚至可能被优化成立即数。

### 易错点 3：字符串字面量不要修改

```c
char *p = "hello";
p[0] = 'H';  // 未定义行为
```

应改为：

```c
char s[] = "hello";
s[0] = 'H';
```

### 易错点 4：指针在哪里，不代表它指向的对象在哪里

```c
int *p = malloc(sizeof(int));
```

- `p` 通常在栈上。
- `*p` 在堆上。

### 易错点 5：`.bss` 不是“不占内存”

`.bss` 通常不占可执行文件实际数据内容，但程序运行时要占内存虚拟地址空间，并在需要时映射到物理页。

### 易错点 6：全局未初始化变量不是随机值

```c
int g;
```

`g` 默认初始化为 0，通常在 `.bss`。

### 易错点 7：普通局部未初始化变量值不确定

```c
void f(void) {
    int x;
}
```

`x` 是自动变量，未初始化时值不确定。

## 17. 判断口诀

可以按这个顺序判断：

```text
先问是不是动态分配：
    malloc/calloc/realloc 得到的对象 -> 堆
否则问是不是普通局部自动变量：
    非 static 的局部变量/数组/参数 -> 栈或寄存器
否则问是不是静态存储期：
    全局变量、static 全局、static 局部 -> .data 或 .bss
        非零初始化 -> .data
        未初始化或零初始化 -> .bss
否则问是不是只读常量/字符串字面量：
    字符串字面量、全局只读 const -> 通常 .rodata
函数代码：
    -> .text
线程局部变量：
    -> TLS，每个线程一份
```

更短的考试版：

```text
代码进 text；
只读常量多进 rodata；
已初始化全局/静态进 data；
未初始化或零初始化全局/静态进 bss；
普通局部变量进 stack；
malloc 出来的对象进 heap；
指针本身看声明位置；
线程局部变量进 TLS。
```

## 18. 从课程知识串起来理解

课程中的内存管理内容可以串成一条主线：

1. C 程序编译、汇编、链接后形成可执行文件。
2. 可执行文件中按 ELF Section 组织，如 `.text`、`.rodata`、`.data`、`.bss`。
3. OS 加载器按 Segment 把程序映射到进程虚拟地址空间。
4. 程序运行后，除了文件中已有的代码和数据，还需要堆、栈、共享库映射区等运行时区域。
5. 每个进程看到的是自己的虚拟地址空间。
6. MMU 通过页表把虚拟地址翻译成物理地址。
7. 多线程共享同一进程的大部分地址空间，但每个线程有自己的栈和寄存器。
8. 因此，判断变量在哪个段，本质上是在判断它的存储期、初始化方式、访问权限和分配方式。

## 19. 最终总结表

| 判断条件 | 存储位置 |
|---|---|
| 函数机器指令 | `.text` |
| 字符串字面量 | `.rodata` |
| 全局/静态变量，非零初始化 | `.data` |
| 全局/静态变量，未初始化或零初始化 | `.bss` |
| 普通局部变量 | 栈，或优化到寄存器 |
| 函数参数 | 栈或寄存器 |
| `malloc` 分配对象 | 堆 |
| 指针变量本身 | 由它自己的声明位置决定 |
| 指针指向对象 | 由被指向对象的来源决定 |
| 全局 `const` | 通常 `.rodata` |
| 局部 `const` | 通常栈或寄存器 |
| `extern` 声明 | 不分配空间，定义处决定 |
| TLS 变量 | 线程局部存储区 |

一句话总括：

> C 变量放在哪个内存区域，核心不是看名字，而是看它的存储期、作用域、初始化方式、是否只读、以及是否动态分配；再通过 ELF 装载和虚拟内存机制映射到进程的虚拟地址空间中。
