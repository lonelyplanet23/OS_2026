# 操作系统 - 第五周第一次课：虚拟存储管理

---

## 课程导入：什么是虚拟存储空间

大家好，今天我们来讲虚拟存储管理。"虚拟"这个词，大家之前已经听过物理地址空间、逻辑地址空间，很多概念长得挺像，容易搞混。我们这里说的虚拟存储空间，本质上就是**对物理存储的一种抽象和仿真**，它不是物理实在，但给程序和用户提供了一个"内存变大了"的假象。

它要解决的核心问题是什么？其实就是一句话：**突破物理内存的容量限制**。

我们原来想：机器装了1G内存，就只能用1G，装不下2G的程序，这个结论对不对？其实这个结论有个前提——你占了内存就永远不释放，占着茅坑不拉屎。如果我们允许内存"时空复用"——一块内存用完了，可以让给别人用，用完再换回来，那是不是就能突破物理限制了？同样一块物理内存，时间上分好几次用，不就能装下更多东西了吗？

主要有两种思路来实现这种时空复用，一种叫**覆盖**，一种叫**交换**。我们一个一个来看。

---

## 第一节：覆盖技术

### 覆盖的基本思想

覆盖是什么意思呢？就是把一个大程序，按照调用关系，划分成一系列独立的**覆盖段**。原则就是：**不会同时用到的代码数据，可以放在同一个覆盖段里，共享同一块内存**。

比如说，运行A的时候不用C，运行C的时候不用A，那A和C就可以组成一个覆盖段。运行哪个就把哪个装进来，后运行的覆盖先运行的，所以叫"覆盖"。

那问题来了：一个覆盖段里放了好几个模块，我们给它分配的覆盖区应该留多大？答案很简单：**等于最大那个模块的大小就够了**。反正同一时间只有一个在内存里，给最大的留位置，其他人都能放得下。

这样做，是不是就相当于把一块内存反复用，在一定程度上就突破了物理内存的限制？一个程序本来比内存大，通过覆盖就能装下运行了。

### 覆盖的例子

我给大家举个例子：主程序要调用四个子程序A、B、C、D，我们分析发现，A和B不会同时调用，C和D不会同时调用。那我们就可以把A和B组成覆盖段1，C和D组成覆盖段2。

假设A是15KB，B是18KB，那覆盖段1我们就分配18KB。C是14KB，D是10KB，覆盖段2就分配14KB。这样四个模块一共占了 `18+14 = 32KB`，如果不覆盖就是 `15+18+14+10 = 57KB`，一下子省出25KB，这就是覆盖的效果。

运行的时候，先装A和C，A用完要调用D的时候，C暂时不用了，就把D装进来覆盖掉C，程序照样正常运行。

### 覆盖的缺点

覆盖解决了问题，但它有个很大的缺点：**对程序员不友好**。程序员写代码的时候，就得自己分析调用关系，划分覆盖段，还要告诉操作系统哪个模块跟哪个模块覆盖，这增加了编程负担。对于复杂程序，分析清楚哪些模块不会同时用到本身就很难，所以现在覆盖技术基本不用了，它更多是作为一个思想，引出我们后面的虚拟存储。

## 第二节：交换技术

交换是另一种思路，跟覆盖不一样。交换是说：把**暂时不用的数据或者整个进程，换到辅存（磁盘）上去，腾出空间给现在要用的进程**。要用到的时候再换回来。

这里我先插一句，大家一定要搞清楚主存和辅存的区别：

- **主存（内存）**：直接跟CPU通过存储总线相连，CPU可以直接访存，速度快，容量小
- **辅存（磁盘）**：CPU不能直接访问，要通过IO控制器、DMA才能把数据搬到主存，速度慢，容量大

交换的目的很明确：主存空间不够了，把那些暂时不用的挪到磁盘去，腾出位置给现在要用的。这样一来，同样的物理内存就能运行更多并发程序，相当于扩充了内存容量。

### 交换和覆盖的关键区别

这里一定要记住：**交换对程序员是透明的**！程序员根本感知不到交换的存在，完全是操作系统自动帮你做的。这跟覆盖需要程序员手动划分完全不一样，这是最大的区别。

交换的优点很明显：它能增加并发程序的数目，把挂起、后台等待的进程换出去，让更多就绪进程跑起来。缺点就是交换本身需要IO，开销比较大，换入换出要花时间。

### 交换中的策略问题

交换不光是机制，还涉及很多策略问题：

**1. 换出谁？**
- 基本原则：换出**最近不会用到**的进程
- 典型：后台进程、正在等待IO的进程，这些进程暂时不用CPU，可以换出去
- 操作系统内核的核心代码和数据一般不换出，因为随时可能要用

**2. 什么时候换出？**
- **积极策略**：一旦发现哪个进程一段时间不用，就主动换出去，提前腾出空间
  - 优点：需要空间时立刻有，不卡
  - 缺点：可能刚换出去又要用到，来回换增加开销
- **懒惰策略**：直到分配内存失败，主存真不够用了，才触发换出
  - 优点：减少不必要的换出
  - 缺点：分配内存的时候可能卡住，用户感觉卡顿（你开大软件的时候卡半天就是这个原因）

交换技术的思想非常重要，**现代虚拟存储就是建立在交换思想的基础上**。

## 第三节：局部性原理 —— 虚拟存储的理论基础

好了，有了覆盖和交换的思想基础，我们来看虚拟存储。虚拟存储能工作，它的理论依据就是**局部性原理**。我再带大家复习一下：

什么是局部性原理？**程序在较短时间内，执行的指令和访问的数据，都集中在一个很小的区域里**。分为两种：

- **时间局部性**：一条指令/一个数据最近被访问过，很快很可能再次被访问（比如循环、变量反复读写）
- **空间局部性**：一个存储单元被访问过，它附近的单元很快很可能被访问（比如数组顺序遍历）

### 案例分析：矩阵乘法为什么循环顺序会影响速度？

我给大家举一个非常经典的例子，帮助大家理解局部性。就是矩阵乘法不同循环顺序，实际运行速度差很多。

我们做C = A × B，两种写法：

**IJK版本（按数学定义写）**：
```c
for (i = 0; i < N; i++)
    for (j = 0; j < N; j++) {
        C[i][j] = 0;
        for (k = 0; k < N; k++)
            C[i][j] += A[i][k] * B[k][j];
    }
```

**KIJ版本（交换循环顺序）**：
```c
for (k = 0; k < N; k++)
    for (i = 0; i < N; i++) {
        r = A[i][k];
        for (j = 0; j < N; j++)
            C[i][j] += r * B[k][j];
    }
```

两个算法结果完全一样，但实际跑大矩阵的时候，速度差一倍！我们实验结果：N很大时，IJK要11秒，KIJ只要5秒。为什么？

**关键就是局部性和Cache Line**：

C语言数组是按行存储的。看KIJ版本的最内层循环，j从0到N-1，访问的是 `B[k][0], B[k][1], B[k][2]...`，这正好是B的第k行连续排列，空间局部性非常好。当你访问 `B[k][0]` 的时候，CPU Cache会把整个Cache Line（比如连续8个或16个元素）都加载进来，后面连续访问就都命中Cache，不用去内存读。

而IJK版本最内层循环是k，访问 `B[k][j]`，每次k都变，跳着访问不同行，空间局部性差，Cache命中率很低，大部分访问都要去主存，自然就慢了。

你看，仅仅换个循环顺序，利用好局部性，速度就翻一倍。这就是为什么理解局部性原理对写出高效代码这么重要。

### 延伸：排序算法的局部性

不止矩阵乘法，排序算法也一样：

- **堆排序**：算法分析上最坏复杂度也是O(n log n)，跟快排一样。但实际跑起来比快排慢很多，为什么？因为堆排序要频繁调整堆，把树深处的节点换到堆顶，访问跨度大，局部性差，Cache命中率低。
- **快速排序**：顺序访问多，局部性好，实际运行更快。
- **STL的sort**：很聪明，小数组用插入排序（O(n²)但局部性好，实际更快），大数组用快排，充分利用局部性原理。

所以记住：算法分析的大O复杂度不是一切，实际运行速度还要看局部性。

## 第四节：虚拟存储管理

好了，回到内存管理。我们之前讲的分区、分页、分段，都是**实存管理**，什么意思？就是要求程序**一次性全部装入内存**才能运行，装不下就不能跑。这样会带来什么问题？

- 如果程序比物理内存大，就跑不了
- 很多程序同时运行，物理内存不够，就不能接纳更多作业

虚拟存储就是来解决这个问题的。**核心思想就是：基于交换和覆盖的思想，程序不用全部一次性装入，只用把现在要用的部分装进来，运行中需要哪部分再调哪部分**。这样，即使程序比物理内存大，也能跑，也能同时运行更多程序。

### 虚拟存储给进程提供了什么？

虚拟存储技术，给**每个进程**提供了：
> 一个大的、一致的、连续的、私有的地址空间

- **大的**：比实际物理内存大得多，容量上限由地址总线位数决定
- **一致的**：每个进程都有这样的地址空间，统一的布局
- **连续的**：程序不用关心自己被分到哪里，看到的是连续空间
- **私有的**：各进程地址空间隔离，一个进程踩内存不会影响别人

从效果上看，就是把**主存和辅存结合起来**，统一管理，让用户感觉好像有一个比实际物理内存大得多的"虚拟内存"能用。对用户来说，你就直接用这个大地址空间，缺什么操作系统自动帮你调，不用你管。

### 虚拟存储的三大特征

我给大家总结一下虚拟存储的特征，非常好记：

| 特征 | 含义 |
|------|------|
| **多次性** | 程序不用一次性全部装入，分多次调入内存 |
| **对换性** | 运行过程中，可以把暂时不用的换出去，需要的换进来 |
| **虚拟性** | 逻辑上扩充容量，给用户提供大容量地址空间，是"虚拟"的 |

四个字总结就是：**按需调页**。用得到的调进来，用不到的放出去。

### 实现方式：虚拟分页最常见

虚拟存储可以基于分页，也可以基于分段：

- **请求分页**：最常用，以页为单位换入换出
- **请求分段**：以段为单位换入换出，段可以动态增长

原理上都可以，实际系统用得最多的是请求分页。页表里面只要加几个标志位，就能实现虚拟存储了：
- 存在位：当前页是否在内存
- 修改位、访问位：帮助页面置换
- 外存地址：不在内存时，到哪里去找

这样，原来的纯分页系统就改成请求分页（虚拟分页）了。

## 第五节：Cache 与 虚拟存储的对比

很多同学会问：Cache也是利用局部性，把常用的数据放在更快的存储里，不常用的放在慢的里面，这跟虚拟存储有什么区别？这里给大家辨析清楚：

| 对比维度 | Cache | 虚拟存储 |
|----------|-------|----------|
| **解决的问题** | **速度差** → CPU比内存快很多，用Cache填补速度差距 | **容量问题** → 物理内存容量不够，用磁盘来扩充 |
| **不命中开销** | 小，大约几十时钟周期 | 大，毫秒级磁盘IO，差了好几个数量级 |
| **存储层次** | CPU <-> 主存之间 | 主存 <-> 辅存之间 |
| **实现者** | 主要由硬件实现 | 主要由操作系统实现 |
| **透明性** | 对OS和程序员都完全透明 | 对应用程序员透明，操作系统要管理 |

二者都用了局部性原理，思想是一样的，就是所在的层次不同，解决的问题不同。

## 第六节：实存管理 vs 虚存管理

我给大家梳理一下整个存储管理的知识框架：

```
存储管理
├─ 实存管理（无交换，一次性装入）
│  ├─ 分区存储（连续分配）
│  ├─ 纯分页（离散分配）
│  ├─ 纯分段
│  └─ 段页式
└─ 虚存管理（引入交换，按需装入）
   ├─ 请求分页
   ├─ 请求分段
   └─ 请求段页式
```

引入交换机制之后，才真正有了虚拟存储。虚拟存储最核心的工作就是三件事：
1. **地址映射**：虚拟地址 → 物理地址（可能在内存，可能在外存）
2. **缺页调入**：访问到不在内存的页，触发缺页中断，从外存调入
3. **页面置换**：内存满了，要把哪个页换出去腾出空间

这些机制都是操作系统来做，对用户透明，用户不用关心内存够不够，操作系统帮你搞定。

## 第七节：进程虚拟地址空间布局

我们以32位X86 Linux为例，看看一个进程的虚拟地址空间长什么样：

32位地址空间总共4GB，典型划分是：

| 地址范围 | 大小 | 用途 |
|----------|------|------|
| `0 ~ 0x08048000` | ~128MB | **禁止区域** → 捕获空指针访问，解引用NULL直接报错 |
| `0x08048000 ~ 3GB` | ~3GB | **用户空间** → 代码段、数据段、BSS、堆、用户栈 |
| `3GB ~ 4GB` | 1GB | **内核空间** → 操作系统内核代码、数据，每个进程都映射 |

为什么要把内核空间映射到每个进程的地址空间里？

因为系统调用的时候，进程要陷入内核执行内核代码，如果内核不在当前地址空间，就要切换地址空间，切换TLB，这代价很高。把内核映射到每个进程地址空间，系统调用直接跳进去就行，不用刷新TLB，性能好很多。

那低地址为什么要留一块禁止？主要就是为了**捕获空指针错误**。程序员经常会写 `if (p == NULL) ...`，但如果一不小心解引用NULL，访问地址0，硬件就能通过页保护直接抓住你，报错退出，而不是读到乱七八糟的数据。

## 第八节：交换区与页表标识

进程的虚拟地址空间，每一页可能在不同地方：

- **在内存**：页表项记录物理页框号
- **在外存**：页表项记录在外存的位置（交换区的哪个槽，或者文件的哪个偏移）

操作系统会给不同类型的段打不同标记：

- **文件映射（file-backed）**：代码段、数据段来自可执行文件，不在内存时直接从文件读
- **匿名映射（anonymous）**：堆、栈这些程序运行才产生的数据，不存在原来的文件，换出时要写到交换区
- **不存在**：还没分配，访问就报段错误

现在Linux系统都是**延迟分配（lazy allocation）**：
- 创建进程的时候，只建立好地址空间布局，不把所有页都装进来
- 第一次访问某一页的时候，触发缺页中断，才真正分配内存或者从文件读进来
- 这样加快了进程启动速度，也节省了内存

## 第九节：请求分页的页表项

我们讲一下，虚拟分页系统中，页表项要增加哪些内容？

原来纯分页只需要物理页框号，现在要加很多标志位：

| 标志位 | 作用 |
|--------|------|
| **存在位（Present）** | 1 = 页在内存，0 = 页在外存 |
| **修改位（Dirty）** | 页被修改过吗？修改过换出时要写回磁盘 |
| **访问位（Accessed）** | 最近被访问过吗？帮助页面置换算法选淘汰页 |
| **保护位** | 读/写/执行权限检查 |

32位地址中，页号占高20位，页内偏移占低12位。页表项只需要存20位物理页框号，剩下的位正好用来放这些标志位，不浪费空间，设计得非常巧妙。
