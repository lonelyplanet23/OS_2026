## 回顾：页式内存管理

### 页式内存管理思想
**破除内存分配"连续性假设"**

### 页表的作用
将虚拟地址空间映射到物理地址空间。

### 32位系统参数
- 地址长度：32位
- 可寻址空间：4GB
- 页内偏移：12位
- 页大小：4KB = 2¹² B
- 页数：4GB / 4KB = 1M = 2²⁰ 页
- 需要1M个页表项（PTE）记录映射关系

---

## 一级页表的问题

**页表的大小与逻辑地址空间大小成正比**。

如果逻辑地址空间很大，则划分的页比较多，页表就很大，占用的物理存储空间大，实现较困难（分配困难）。

### 示例
**32位系统**：
- 页面大小4KB
- 页表项4字节
- 每个进程页表大小 = 2²⁰ × 4B = 4MB（连续物理内存！）

**64位系统更可怕**：
- 2⁶⁴ / 2¹² = 2⁵² 页
- 页表大小 = 2⁵⁴ B = 16 PB 的连续物理内存！

---

## 多级页表

### 解决问题的方法
1. **多级页表**：继续破除页表存储的连续性假设
2. **动态调入页表**：只将当前需用的部分页表项调入内存，其余的需用时再调入

### 32位二级页表
- 32位地址空间
- 4K页面大小
- 地址结构：**10位 + 10位 + 12位**
  - 高10位：页目录索引（PDE）
  - 中10位：页表索引（PTE）
  - 低12位：页内偏移
- 共需：1K个页目录项 + 1M个页表项 = 约4MB

## 页目录自映射

### 核心思想
让页目录自己映射自己，形成一个"自引用"的特殊页目录项。

### 页表空间大小计算
- 每个页表项4字节
- 1M个页表项需要：1M × 4B = **4MB**
- 所以整个页表占用的虚拟地址空间大小就是4MB

### 页表分页存储
- 4MB页表也要分页存储
- 共需要：4MB / 4KB = **1024个页面**存储（页表页）
- 每一页中存储：4KB / 4B = 1024项页表项
- 每1个页表页对应：1024 × 4KB = **4MB内存**

---

## 自映射的实现原理

### 什么是自映射？
在页目录中，将其中一个PDE（页目录项）指向页目录自身的物理地址。

### 效果
通过这个特殊的PDE，可以用虚拟地址访问所有页表本身！

### 自映射的作用
1. 操作系统可以**用虚拟地址读写页表**（不需要物理地址）
2. 页表本身也被纳入虚拟地址空间管理
3. 简化页表的创建、修改和销毁操作

### 典型实现
- 通常将自映射放在虚拟地址空间的最高端
- 例如：在3G以上的内核空间分配4MB作为页表的虚拟地址区域
- 这样OS可以直接用指针访问任何页表项

---

## 自映射的访问方式

### 访问页目录本身
```
PD index = 自映射项的索引
PT index = 自映射项的索引
offset = 要访问的PDE在页目录中的偏移
```
这样组合出的虚拟地址，可以访问到页目录自身的内容。

### 访问某个页表页
```
PD index = 自映射项的索引
PT index = 目标页表对应的PDE索引
offset = 目标PTE在该页表中的偏移
```

---

## 自映射的优势

| 优势 | 说明 |
|------|------|
| **简化页表操作** | OS用普通指针就能读写页表，不需要切换到物理地址 |
| **统一内存管理** | 页表本身也作为虚拟内存的一部分被管理 |
| **减少TLB刷新** | 页表区域通常在内核空间，进程切换不需要刷新这部分TLB |
| **易于扩展** | 动态分配和释放页表页变得简单 |

## 本章总结

1. **一级页表问题**：太大，需要连续物理内存
2. **多级页表解决**：破除连续性假设，分层管理
3. **页表管理问题**：OS需要用虚拟地址访问页表本身
4. **自映射**：通过让页目录指向自己，实现用虚拟地址访问所有页表
5. **4MB页表区域**：32位系统下，整个页表占4MB虚拟地址空间
