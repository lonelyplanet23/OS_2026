## 几个算法的共性问题

无论是软件解法（如Peterson）还是硬件（如TSL或XCHG）解法都是正确的，它们都有一个共同特点：当一个进程想进入临界区时，先检查是否允许进入，若不允许，则该进程将原地等待，直到允许为止。

**两个问题**：
1. **忙等待**：浪费CPU时间
2. **优先级反转**：低优先级进程先进入临界区，高优先级进程一直忙等

---

## 解决之道

**忙等 → 阻塞**

- **Sleep**：进程主动放弃CPU，进入阻塞状态
- **Wakeup**：当条件满足时，唤醒阻塞的进程

**例子：银行排队**
- 忙等：看到队很长，坚持排队
- 阻塞：看到队很长，先回家歇会儿（sleep），有空柜台了，大堂经理（scheduler）电话通知再过来（wakeup）

## 信号量机制

- 1965年由 **Dijkstra** 提出
- P(S)，V(S)操作：P、V分别是荷兰语的 test(proberen) 和 increment(verhogen)
- 基本思路是使用一种新的变量类型（semaphore）
- 信号量只能通过初始化和两个标准的原语来访问，作为OS核心代码执行，不受进程调度的打断

---

## 经典信号量机制

```c
P(S):
    while S <= 0 do skip  // 忙等
    S := S - 1;

V(S):
    S := S + 1;
```

---

## 计数信号量机制

```c
Type semaphore = record
    value : integer;    // 信号量值
    L : list of process; // 等待队列
end

Procedure P(S)
    var S : semaphore;
    begin
        S.value := S.value - 1;
        if S.value < 0 then
            block(S.L);  // 阻塞，加入等待队列
    end

Procedure V(S)
    var S : semaphore;
    begin
        S.value := S.value + 1;
        if S.value <= 0 then
            wakeup(S.L); // 唤醒等待队列中的一个进程
    end
```

---

## 信号量的使用规则

- **必须置一次且只能置一次初值**
- 只能由P、V操作来改变

---

## 物理意义

| S.value 的值 | 含义 |
|--------------|------|
| > 0 | 表示可用资源的个数 |
| < 0 | 表示等待该资源的进程个数 |

**P操作（wait）**：分配资源
- 如果无法分配则阻塞

**V操作（signal）**：释放资源
- 如果有等待进程则唤醒

## 信号量的应用

### 1. 互斥
利用信号量实现进程互斥
- **S = 1**（二元信号量/互斥信号量）

```c
P(mutex);   // 进入临界区前
临界区
V(mutex);   // 退出临界区后
```

### 2. 同步
利用信号量实现进程同步
- **S = 0**（同步信号量，表示某个条件是否满足）

例如：描述进程执行的前趋/后继关系

```c
// 进程A                // 进程B
...                     P(S);  // 等待A完成
V(S);  // A完成后发信号   ...
```
