## 课程导入：缺页错误处理流程回顾

上次课我们讲了调页机制，这是虚拟存储的基石。要实现虚拟存储，让用户感觉在用一个比实际物理内存大得多的地址空间，就必须依靠缺页错误处理机制。

今天我们再一起梳理一遍整个缺页错误（page fault）的处理流程，搞清楚每个步骤到底谁在做什么——哪些是硬件做的，哪些是软件（操作系统）做的。

---

## 第一节：缺页错误完整处理流程

当CPU执行一条指令，要访问一个虚拟地址时，过程是这样的：

### 步骤分解

| 步骤 | 操作 | 谁来做 | 说明 |
|------|------|--------|------|
| 1 | CPU取出虚拟地址，尝试通过MMU/TLB翻译为物理地址 | **硬件** | TLB命中直接得到物理地址 |
| 2 | 如果页表中发现该页**Present位为0**（不在内存） | **硬件** | 自动触发page fault异常，进入内核态 |
| 3 | 保存用户进程上下文（寄存器等） | **硬件+OS** | 切换到内核栈，保存所有寄存器状态 |
| 4 | OS根据异常信息判断缺页原因，从外存读取所需页面 | **OS（软件）** | 查找该页在外存的位置，发起IO读取 |
| 5 | 分配物理页框，更新页表项 | **OS（软件）** | 设置Present位=1，填入物理页框号 |
| 6 | 更新TLB | **OS/硬件** | 刷新TLB，加入新映射 |
| 7 | 恢复用户进程上下文，**重新执行产生缺页的那条指令** | **OS恢复+硬件** | 注意：是**重新执行**这条指令，不是执行下一条 |

### 关键考点：为什么要"重新执行原指令"？

因为缺页发生在指令**执行过程中**，指令没有完成执行。处理完缺页后，必须从头开始重新取指、译码、执行，才能正确完成这次访存。

> 对比：一般性中断处理完是回到**下一条**指令执行。缺页错误不一样，必须回到**当前这条**指令重新执行。这是常考点。

## 第二节：硬件软件分工辨析

经常考：缺页处理流程中，哪些步骤由硬件完成，哪些由操作系统完成？

| 操作 | 执行者 | 原因 |
|------|--------|------|
| 查页表/TLB | 硬件（MMU） | 地址翻译需要速度，硬件更快 |
| 检测Present位，触发page fault | 硬件 | 这是CPU异常机制，硬件自动检测 |
| 保存上下文、切换栈 | 硬件+OS | CPU自动完成一部分，OS完成保存 |
| 从外存读页、分配页框 | OS | 这是操作系统存储管理的职责 |
| 更新页表 | OS | OS知道分配情况，修改页表 |
| 恢复上下文、重新执行 | OS+硬件 | OS恢复，硬件执行 |

### 上下文保存存在哪？

缺页发生，陷入内核时，必须保存用户进程的寄存器上下文。保存在哪里？**保存在内核栈**，不是用户栈。

为什么？因为：
1. 如果直接保存在用户栈，可能覆盖用户原来的数据
2. 用户栈本身可能不在内存，还可能继续缺页
3. 内核有自己独立的栈，专门用来保存异常/中断上下文

所以：trap进入内核时，要**切换栈指针到内核栈**，然后把寄存器保存在内核栈上。返回时再从内核栈恢复，切换回用户栈。

## 插播：C语言内存布局基础（帮助实验）

在讲页面置换算法之前，我们插播一些C语言内存布局的基础知识，这对大家做实验很有帮助。

### 1. 结构体对齐

来看这个例子：

```c
// S1
struct S1 {
    char c;    // 1字节
    short s;   // 2字节
    int i;     // 4字节
};

// S2
struct S2 {
    char c;    // 1字节
    int i;     // 4字节
    short s;   // 2字节
};
```

两个结构体成员一样，只是顺序不同，`sizeof`结果一样吗？**不一样**。S1是8字节，S2是12字节。

为什么？因为**对齐要求**：
- int（4字节）必须从**4字节对齐**的地址开始
- 在S1中：c(1) → 填充1字节 → s(2) → i(4) → 总共 1+1+2+4 = 8字节
- 在S2中：c(1) → 填充3字节 → i(4) → s(2) → 总共 1+3+4+2 = 12字节

**为什么需要对齐？**
这是硬件限制：很多CPU架构要求**对N字节数据的访问必须从N字节对齐的地址开始**。如果不对齐，要么访存出错，要么性能下降。编译器自动帮你插入填充字节，满足对齐要求。

### 2. C语言可变参数（printf）

printf的`...`是怎么实现的？这依赖于C语言的调用约定：

- C语言参数从**右向左**依次入栈
- 第一个参数一定是格式字符串`format`
- printf解析format，知道有几个参数，然后从栈上依次取出后面的参数

`va_list`、`va_start`、`va_arg`这些宏，本质上就是帮你计算栈上参数的地址，一个一个取出来。

如果format里写了三个`%d`，但你只传了两个参数，会发生什么？**编译能过，但运行时printf会越界访问栈**，读出垃圾数据，可能导致崩溃。这是常见的C语言错误。

## 第三节：页面置换算法

当发生缺页时，如果内存还有空闲页框，直接分配就行了。如果内存已经满了，就要选一个现有页面换出去，给新页腾位置。这就是**页面置换**。选哪个页面换出去？这就是置换策略问题。

我们讲几种经典算法：

---

### 1. OPT（最优置换算法）

**思想**：置换掉**未来最久才会被使用**，或者**永远不会被使用**的页面。

- 优点：缺页次数**最少**，是理论下界
- 缺点：**无法实现**，因为你运行时不知道未来哪个页面最久不用
- 用途：作为理论比较的基准，评价其他算法

**关键结论**：OPT算法是理论最优，但不可实现，只用于比较。

---

### 2. FIFO（先进先出）

**思想**：置换掉在内存中**驻留时间最长**的页面。先来的先走。

- 实现：维护一个队列，新页进来队尾，淘汰队首
- 优点：实现极其简单
- 缺点：性能差，可能把经常访问的常用页面置换出去

#### Belady异常（重点）

**现象**：给进程分配的页框数增多了，**缺页率反而升高**。这是反直觉的。

**举例**：页面走向 `1 2 3 4 1 2 5 1 2 3 4 5`
- 分配3个页框：9次缺页
- 分配4个页框：反而10次缺页 → Belady异常

**为什么会发生**：因为FIFO只看进入时间，不看访问频率，它的置换顺序和访问频率脱节。

**哪些算法会出现**：FIFO以及FIFO的一些改进算法都可能出现。OPT算法不会。

---

### 3. Clock（时钟）算法 / Second Chance（第二次机会）

这是FIFO的改进，思路：给每个页面一个**访问位**（reference bit）。

#### Second Chance 算法

- 维护一个FIFO队列
- 要置换时，看队首页面：
  - 如果访问位 = 1 → 说明这个页面最近被访问过，**给它第二次机会**，清除访问位，放到队尾
  - 如果访问位 = 0 → 置换它

#### Clock 算法（环形队列改进）

- 把队列改成**循环队列**，用一个指针顺时针扫描
- 指针指向谁，就检查谁：
  - 访问位 = 1 → 清访问位，指针前进
  - 访问位 = 0 → 置换它，指针前进

- 优点：实现简单，性能比纯FIFO好很多
- 缺点：还是可能Belady异常
- 实际系统中很常用

---

### 4. LRU（最近最少使用）

**思想**：置换掉**最近最少使用**的页面。这符合局部性原理——最近用过的页面很快还会用，最近没用的很快也不会用。

- 优点：性能好，很接近OPT
- 缺点：实现复杂，开销大

**为什么实现难**：
- 需要精确记录每个页面最近被访问的时间顺序
- 每次访问都要更新数据结构，开销大
- 纯软件实现会导致内存访问放大，性能下降

**近似实现**：
- 位矩阵法（Clock近似）：用一个二进制矩阵，访问时设置对应位，最后找全0的行就是最少使用
- 硬件辅助：把访问位放在PTE/TLB中，硬件自动更新，OS只读不用每次都写

---

### 算法对比总结

| 算法 | 思想 | 实现 | 缺页率 | Belady异常 | 实际使用 |
|------|------|--------|----------|------------|----------|
| OPT | 换未来最久不用 | 不可实现 | 最低 | 不会 | 仅作为基准 |
| FIFO | 换最早进来 | 极简单 | 高 | 会 | 很少用 |
| Clock/Second Chance | 给访问过的第二次机会 | 简单 | 中等 | 可能会 | **常用** |
| LRU | 换最近最少用 | 复杂 | 低（接近OPT） | 不会 | 用硬件辅助会用 |

## 第四节：计算题做题方法（考试重点）

考试一定会给你一个页面走向，让你计算不同算法的缺页次数。步骤如下：

1. **初始**：页框全空，所以刚开始填入每个页面都算缺页
2. **命中**：页面已经在内存 → 不缺页，更新访问位（如果算法需要）
3. **缺页**：页面不在内存 → 缺页计数+1，如果有空闲页框直接填入，否则按算法选择一个置换出去
4. **最后**：统计总缺页次数 = 缺页计数

### 举例：OPT vs LRU 对比

页面走向：`1 2 3 4 1 2 5 1 2 3 4 5`，分配3个页框

**OPT**：
- 每次缺页时，看当前内存中三个页面，哪个**未来最远才用**，就换哪个
- 结果缺页次数比FIFO少，比LRU少

**LRU**：
- 每次缺页时，看当前内存中三个页面，哪个**最近最久没被访问**，就换哪个
- 是"往回看"历史，不是"向前看"未来

**关键区别**：
- OPT：**向前看**未来 → 不可实现
- LRU：**向后看**历史 → 可实现，用历史预测未来
