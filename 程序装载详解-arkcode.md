## 一、整体流程概述

从源代码到运行，一个程序需要经历三个关键步骤：

```
编译 (Compile) → 链接 (Link) → 装载 (Load)
```

| 阶段 | 输入 | 输出 | 核心工作 |
|------|------|------|----------|
| 编译 | C 源代码 | 汇编代码 | 语法分析、生成汇编指令 |
| 汇编 | 汇编代码 | 目标文件 (.o) | 生成 ELF 格式可重定位目标文件 |
| 链接 | 多个目标文件 | 可执行文件 | 合并段、符号解析、重定位 |
| 装载 | 可执行文件 | 运行中的进程 | 建立地址空间、映射文件、启动执行 |

## 二、程序的段结构

### 1. 程序在磁盘（可执行文件）中的分段

一个程序在编译后会被分成不同的段，每个段有不同的作用：

| 段名 | 内容 | 权限 | 特点 |
|------|------|------|------|
| `.text` | 程序代码（机器指令） | 只读、可执行 | 大小在编译时确定 |
| `.data` | 已初始化的全局变量和静态变量 | 可读、可写 | 占用可执行文件空间 |
| `.bss` | 未初始化的全局变量和静态变量 | 可读、可写 | **不占用可执行文件空间**，装载时由系统清零 |
| `.rodata` | 只读数据（如字符串常量） | 只读 | 和 .text 放一起 |
| `.plt` / `.got` | 动态链接相关 | 不同 | 用于延迟绑定 |

### 2. 运行时的动态分配区域

除了上面编译确定的段，程序运行时还有两个动态区域：

| 区域 | 用途 | 生长方向 |
|------|------|----------|
| **堆 (Heap)** | `malloc`/`new` 动态分配的内存 | 向上（高地址）生长 |
| **栈 (Stack)** | 局部变量、函数参数、返回地址 | 向下（低地址）生长 |

### 例子（来自您的 Notes）

```c
int global_init = 42;     // .data 段
int global_uninit;        // .bss 段
const int const_global = 100;  // .rodata 段

int main() {
    int a = 10;                    // 栈
    static int static_init = 20;   // .data 段
    static int static_uninit;      // .bss 段
    printf("Hello World");         // "Hello World" 字符串在 .rodata
    int* p = malloc(sizeof(int)); // p 本身在栈，分配的内存在堆
    return 0;
}
```

## 三、ELF 文件格式

ELF (Executable and Linkable Format) 是 Linux 系统下标准的可执行文件格式。

### 1. 两种视图

| 视图 | 单位 | 使用阶段 | 说明 |
|------|------|----------|------|
| **链接视图** | Section（节） | 编译链接阶段 | 每个节有不同类型（.text, .data 等） |
| **执行视图** | Segment（段） | 装载运行阶段 | 多个属性相同的节合并为一个段 |

### 2. ELF 文件结构

```
+-------------------+
|   ELF Header      |  文件头，描述整体结构
+-------------------+
|   Program Header  |  程序头表（装载需要）┐
|   Table           |                      ├ 可选？
+-------------------+                      |
|                   |                      |
|   Section 1       |                      |
|   Section 2       |  各个节（链接需要）  链接阶段用
|   ...             |                      |
|                   |                      |
+-------------------+                      |
|   Section Header  |                      |
|   Table           |  节头表（链接需要）┘
+-------------------+
```

### 3. ELF 头关键字段

```c
typedef struct {
    unsigned char e_ident[16];  // 魔数 0x7fELF + 其他标识
    uint16_t      e_type;       // 文件类型 (可执行/可重定位/共享)
    uint16_t      e_machine;    // 机器架构 (MIPS/x86/...)
    uint32_t      e_entry;      // 入口地址 (_start)
    uint32_t      e_phoff;      // 程序头表偏移
    uint32_t      e_shoff;      // 节头表偏移
    uint16_t      e_phnum;      // 程序头表项数
    uint16_t      e_shnum;      // 节头表项数
} Elf32_Ehdr;
```

### 4. 程序头表项（关键！加载器用）

```c
typedef struct {
    uint32_t  p_type;    // 段类型
    uint32_t  p_offset;  // 段在文件中的偏移
    uint32_t  p_vaddr;   // 段的虚拟地址
    uint32_t  p_paddr;   // 物理地址（忽略）
    uint32_t  p_filesz;  // 段在文件中的大小
    uint32_t  p_memsz;   // 段在内存中的大小
    uint32_t  p_flags;   // 权限标志
    uint32_t  p_align;   // 对齐要求
} Elf32_Phdr;
```

**关键说明**：
- `p_type = PT_LOAD`：表示这个段需要被加载到内存
- `p_memsz > p_filesz`：多出的部分是 BSS，需要清零
- `p_flags`：R(读)、W(写)、X(执行)

## 四、程序链接过程

### 1. 静态链接步骤

#### 步骤 1：空间与地址分配
- 扫描所有输入目标文件，获得各个段的长度、对齐要求
- 将相同属性的段合并在一起
- 计算每个段在最终地址空间中的位置

#### 步骤 2：符号解析与重定位
- 每个目标文件有一个符号表，记录所有定义和引用的符号
- 链接器建立一个全局符号表
- 对于每个未定义的符号引用，查找它的定义位置
- 修改所有对符号的引用，填入正确的地址（这就是**重定位**）

### 2. 重定位条目

重定位条目告诉链接器哪些位置需要修改：

```c
typedef struct {
    Elf32_Addr r_offset;  // 需要修改的地址偏移
    Elf32_Word r_info;    // (符号索引 << 24) | 重定位类型
} Elf32_Rel;
```

**MIPS 常见重定位类型**：
- `R_MIPS_26`：26位跳转地址，用于 `jal` 指令
- `R_MIPS_HI16`：地址高16位，用于 `lui`
- `R_MIPS_LO16`：地址低16位，用于 `addi`

### 3. 静态链接 vs 动态链接

| 特性 | 静态链接 | 动态链接 |
|------|----------|----------|
| 链接时机 | 编译时 | 运行时 |
| 可执行文件大小 | 大（包含所有库代码） | 小（只包含引用） |
| 内存占用 | 每个进程一份库代码 | 多个进程共享一份 |
| 启动速度 | 快 | 稍慢（需要加载动态链接器） |
| 更新库 | 需要重新链接 | 只需替换共享库 |

## 五、程序装载完整流程

这是从 shell 输入命令到程序开始运行的全过程。

### 第一步：创建进程 - fork()

```
输入命令 → shell
    ↓
shell 调用 fork()
    ↓
    ┌─────────────────┴─────────────────┐
    ↓                                  ↓
shell (父进程，等待子进程)        shell (子进程)
                                     ↓
                               调用 execve()
```

### 第二步：装载程序 - execve() 系统调用

`execve()` 的函数原型：
```c
int execve(const char *filename, char *const argv[], char *const envp[]);
```

**execve() 内部执行步骤**：

#### 📌 步骤 1：验证文件格式
1. 打开文件
2. 读取 ELF 头，检查魔数 `0x7f 'E' 'L' 'F'`
3. 如果不是 ELF，检查是否是脚本（以 `#!` 开头）

#### 📌 步骤 2：读取程序头表
遍历所有程序头表项，找出所有类型为 `PT_LOAD` 的段，这些是需要加载的。

#### 📌 步骤 3：创建虚拟地址空间
1. 销毁子进程原来的地址空间（来自 shell）
2. 为新程序创建新的页表
3. **此时还没有分配物理内存！** 只是建立了虚拟地址到文件的映射关系

#### 📌 步骤 4：映射每个 LOAD 段到虚拟地址

对于每个 `PT_LOAD` 段：
```
虚拟地址范围: [p_vaddr, p_vaddr + p_memsz)
权限: R/W/X 根据 p_flags 设置
映射: 文件的 [p_offset, p_offset + p_filesz) 映射到虚拟地址开头
      剩下的 (p_memsz - p_filesz) 映射到匿名页（需要清零）
```

**关键**：这只是建立映射，并没有真正把文件内容读到内存！真正读发生在缺页异常。

#### 📌 步骤 5：处理特殊段

- **BSS 段**：`p_memsz > p_filesz`，多出部分标记为匿名页，第一次访问时分配物理页并清零
- **动态链接**：如果有 `.interp` 段，记录动态链接器的路径

#### 📌 步骤 6：创建并初始化用户栈

栈通常在虚拟地址空间的高地址，大小一般是 8MB：
```
栈顶 (低地址) ← $sp
    ↓
+----------------+ ← 栈底 (高地址)
|   envp[]       | 环境变量指针数组
+----------------+
|   argv[]       | 参数指针数组
+----------------+
|   argv[0]      | 参数字符串
|   ...          |
|   envp[0]      | 环境变量字符串
+----------------+
```

`argc` 放在栈最顶端，`argv[0]` 在 `argc` 下面。

#### 📌 步骤 7：设置寄存器，准备启动

1. 设置程序计数器 PC = `e_entry`（ELF 头中的入口地址，通常是 `_start`）
2. 设置栈指针 SP 指向用户栈顶
3. 将 `argc` 放入 `$a0`，`argv` 放入 `$a1`（MIPS 约定）
4. 跳转到用户态，从 PC 开始执行

## 六、从 _start 到 main

程序不是从 `main` 开始执行，而是从 `_start` 开始！`_start` 是链接器提供的入口点。

### _start 的主要工作（MIPS 伪代码）

```assembly
_start:
    # 1. 从栈中取出参数
    lw      $a0, 0($sp)         # argc
    addi    $a1, $sp, 4         # argv

    # 2. 初始化 C 运行库
    jal     __libc_init         # 初始化 malloc、stdio 等
    jal     __cxa_initialize    # 初始化 C++ 全局对象（如果有）

    # 3. 调用 main 函数
    jal     main

    # 4. main 返回，退出程序
    move    $a0, $v0             # main 的返回值作为 exit 参数
    jal     exit                 # exit 系统调用，不会返回
```

### 为什么需要 _start？
- `main` 是 C 语言约定的入口，不是操作系统约定的
- 需要先初始化 C 库（printf、malloc 才能工作）
- 需要处理 main 返回后的清理工作

## 六、动态链接装载

### 1. 动态链接的基本概念

- **共享库 (Shared Library)**：`.so` 文件，被多个进程共享
- **动态链接器 (Dynamic Linker)**：也叫 `ld.so`，负责加载共享库和解析符号
- **延迟绑定 (Lazy Binding)**：用到函数时才绑定地址，加快启动速度

### 2. 动态链接装载步骤

当可执行文件依赖动态链接库时：

#### 步骤 1：内核识别动态链接
- 内核读取 ELF 的 `.interp` 段，得到动态链接器路径（如 `/lib/ld-linux.so.2`）
- 内核加载动态链接器到地址空间
- 将 PC 设置为动态链接器的入口，把控制权交给它

#### 步骤 2：动态链接器工作
1. **自举**：动态链接器对自己进行重定位
2. 读取可执行文件的 `.dynamic` 段，获得依赖的共享库列表
3. **加载所有依赖的共享库**，依次映射到地址空间
4. **重定位**：对可执行文件和共享库中所有需要重定位的条目进行处理
5. **符号解析**：解决跨共享库的符号引用

#### 步骤 3：跳转到程序
动态链接器完成工作后，跳转到可执行文件的 `_start`，程序开始正常执行。

### 3. PLT 和 GOT 机制（延迟绑定）

**PLT**：Procedure Linkage Table，代码段中的跳板表
**GOT**：Global Offset Table，数据段中存放真实地址的表

| 表 | 位置 | 内容 | 可写 |
|----|------|------|------|
| PLT | .text | 跳转指令 | 否 |
| GOT | .got.plt | 函数真实地址 | 是 |

**第一次调用函数**：
```
调用 func → PLT[func] → GOT[func] 现在指向 PLT 的下一条指令
           → 动态链接器查找 func 真实地址 → 写入 GOT[func]
           → 跳转到真实地址
```

**后续调用**：
```
调用 func → PLT[func] → GOT[func] 已经是真实地址
           → 直接跳转到真实地址
```

这样，只有第一次调用有开销，之后都是直接跳转。

## 七、缺页异常与按需调页

### 1. 什么是按需调页？

装载程序时，内核并不把整个可执行文件读到内存，只建立了虚拟地址映射。当 CPU 第一次访问某个页面时，触发缺页异常，内核才分配物理内存，从文件读取内容。

**好处**：
- 加快程序启动速度
- 节省物理内存（不用的代码永远不会被加载）

### 2. 缺页异常处理流程

```
CPU 执行指令，访问虚拟地址 X
    ↓
MMU 查询页表 → P = 0 (页不存在)
    ↓
CPU 触发缺页异常 → 进入内核
    ↓
内核缺页处理程序：
    ├─ 判断访问是否合法？
    │   ├─ 非法 → 发送 SIGSEGV → 程序崩溃（段错误）
    │   └─ 合法 → 继续
    ├─ 分配一个物理页框
    ├─ 从文件读取对应内容到物理页
    │   （如果是匿名页如 BSS，则清零）
    ├─ 更新页表：设置 P = 1，填入物理页号
    └─ 返回用户态，重新执行那条指令
```

### 3. 几种缺页类型

| 缺页原因 | 处理方式 |
|----------|----------|
| 代码段访问 | 从可执行文件读取 |
| BSS 访问 | 分配物理页，清零 |
| 堆访问 | 分配匿名页，清零 |
| 栈访问 | 分配匿名页，清零 |
| 非法地址访问 | 发送 SIGSEGV |

## 八、进程地址空间布局（32位）

```
高地址 0xC0000000
    ┌─────────────────┐
    │  内核空间        │  内核代码、数据
    ├─────────────────┤
0xBFFFFFF
    │  环境变量        │
    │  参数区域        │
    ├─────────────────┤
    │  栈 (Stack)      │ ← 向下生长
    │                 │
    │        ...       │  空洞（不用）
    │                 │
    │  堆 (Heap)       │ ↑ 向上生长
    ├─────────────────┤
    │  .bss            │
    ├─────────────────┤
    │  .data           │
    ├─────────────────┤
    │  .text (代码)     │
    ├─────────────────┤
0x08048000
    │  保留区域        │
低地址 0x00000000
```

**要点**：
- 栈在高地址，向下生长
- 堆在低地址，向上生长
- 中间有很大的空洞，可以容纳堆和栈的动态增长

## 九、关键概念总结

### 名空间 vs 地址空间 vs 存储空间

| 概念 | 说明 |
|------|------|
| **名空间** | 程序中符号（变量名、函数名）的集合，编译器处理 |
| **地址空间** | 程序运行使用的逻辑地址范围 |
| **存储空间** | 实际存放程序和数据的物理内存 |

### 重定位分类

| 重定位 | 时机 | 方式 | 特点 |
|--------|------|------|------|
| **静态重定位** | 装载时完成 | 加载器修改地址 | 运行快，但程序不能移动 |
| **动态重定位** | 运行时完成 | MMU 硬件转换 | 程序可以在内存中移动，现代系统都用这个 |

### 静态分配 vs 动态分配

| 分配方式 | 地址确定时机 | 特点 |
|----------|--------------|------|
| **静态分配** | 装载时确定所有地址 | 不能运行时申请附加空间 |
| **动态分配** | 运行时可以申请 | 支持 `malloc` 等动态分配 |

## 装载过程关键步骤回顾

```
shell → fork() → 子进程 → execve()
                   ↓
             读 ELF 头
                   ↓
             读程序头表
                   ↓
             创建虚拟地址空间
                   ↓
             映射 PT_LOAD 段
                   ↓
             设置用户栈 (argc, argv, envp)
                   ↓
             [如果是动态链接 → 加载 ld.so，把控制权交给它]
                   ↓
             设置 PC = _start → 进入用户态
                   ↓
             _start → 初始化 libc → 调用 main()
                   ↓
             程序开始运行！
```

---

**核心思想**：现代操作系统采用**虚拟地址空间 + 动态链接 + 按需调页**，实现了：
1. 程序隔离（每个进程有独立地址空间）
2. 内存高效利用（共享库、按需调页）
3. 程序可以比物理内存更大（交换机制）
