### 高频考点详解

#### 1. 内核态与用户态
- **核心目的**：保护操作系统核心资源，防止应用程序随意访问关键硬件
- **权限差异**：内核态可执行特权指令，用户态只能执行非特权指令

#### 2. 微内核 vs 宏内核
- **宏内核（Monolithic Kernel）**：所有OS功能都运行在内核态，体积大，性能高
- **微内核（Microkernel）**：仅最基本功能在内核态，其他功能在用户态，**占用空间更小，更易扩展，但系统调用开销大**

#### 3. 批处理系统特点
- **优点**：CPU利用率高，系统吞吐量大
- **缺点**：**失去了交互性**，用户无法与作业交互

#### 4. 动态重定位
- 硬件在**每次内存访问时**动态实现地址转换
- 需要基址寄存器支持
- 优点：程序可在内存中移动，便于内存紧凑

### 高频考点详解

#### 1. 多级页表计算（必考！）
- **32位系统 + 4KB页面**：
  - 页内偏移：12位
  - 1级页表：页表项数 = 2^(32-12) = 2^20 = 1M个
  - 无论分配多少内存，1级页表都需要完整的1M个PTE

- **多级页表优势**：减少页表占用内存（只需分配实际用到的页表页）
- **多级页表劣势**：级数越多，平均访问内存时间越长（需要多次查页表）

#### 2. 页面置换算法
| 算法 | 特点 | 关键点 |
|------|------|--------|
| **FIFO** | 简单，选择最早进入的页面淘汰 | 存在Belady现象 |
| **LRU** | 选择最久未使用的页面淘汰 | 性能好但实现开销大 |
| **CLOCK** | 给每个页面设置使用位，循环扫描 | FIFO的改进，近似LRU |
| **二次机会** | 检查使用位和修改位 | CLOCK改进版 |
| **Aging** | 记录每个页面的"年龄" | LRU的高效软件近似 |
| **工作集** | 追踪进程最近访问的页面集合 | 驱逐不在工作集中的页面 |
| **WSClock** | 结合工作集和CLOCK思想 | 高效实用的算法 |

- **Belady现象**：增加页框数时，缺页中断次数反而增加的异常现象（仅FIFO存在）

#### 3. 段页式地址转换流程
```
虚拟地址 → 虚拟段号 → 段表 → 页表基地址 → 虚拟页号 → 页表项 → 物理页框号 → 物理地址
                                                         ↓
                                                    页内偏移
```

#### 4. 覆盖 vs 交换
| 特性 | 覆盖（Overlay） | 交换（Swapping） |
|------|----------------|------------------|
| **范围** | 同一程序/作业内部 | 不同作业/程序之间 |
| **实现者** | 应用程序员（编译器辅助） | 操作系统 |
| **透明度** | **对程序员不透明** | 对程序员透明 |
| **目的** | 运行大于内存的程序 | 提高内存利用率和多道程序度 |

#### 5. 抖动（Thrashing）
- **定义**：进程并发水平上升，常驻集减小，缺页率不断上升，频繁调页导致系统开销增大
- **特征**：CPU利用率很低，I/O设备繁忙
- **解决方法**：
  - 减少多道程序度（终止部分进程）
  - 增加物理内存
  - 采用工作集模型控制并发

#### 6. 可变分区分配算法
- **最佳适配（Best Fit）**：选择可容纳作业的**最小**空闲区 → 产生大量外部碎片
- **最坏适配（Worst Fit）**：选择可容纳作业的**最大**空闲区 → 产生的碎片较大但数量少
- **首次适配（First Fit）**：从头开始找第一个满足的
- **下次适配（Next Fit）**：从上次位置继续找

#### 7. 页表自映射计算技巧
- 一级页表起始虚拟地址 = 页表映射基址的高10位左移22位
- 自映射表项虚拟地址 = 一级页表基址 + 页表索引 × 4

### 高频考点详解

#### 1. 进程 vs 线程
| 特性 | 进程 | 线程 |
|------|------|------|
| **资源分配** | 资源分配基本单位 | 不拥有独立资源（共享进程资源） |
| **调度执行** | 不是调度基本单位 | CPU调度基本单位 |
| **地址空间** | 独立地址空间 | 共享进程地址空间 |
| **上下文切换** | 开销大 | 开销小 |
| **栈** | 独立栈 | **各自拥有独立栈** |
| **寄存器上下文** | 独立 | 独立 |

- **联系**：一个进程可以拥有多个线程，一个线程只能属于一个进程
- **多线程优势**：更好释放多核性能，提高并发度
- **注意**：多线程**不总是**能获得更好性能（同步开销、上下文切换开销）

#### 2. 进程上下文切换步骤
1. **陷入内核**：必须从用户态进入内核态
2. **保存上下文**：保存当前进程的寄存器上下文（PC、PSW、通用寄存器等）到PCB
3. **切换地址空间**：更新页表基址寄存器，TLB失效
4. **选择新进程**：调度算法选择下一个要运行的进程
5. **恢复上下文**：恢复新进程的寄存器上下文
6. **返回用户态**：跳转到新进程的程序计数器

> **重要**：进程上下文切换**一定会**陷入内核，但陷入内核**不一定会**导致进程切换（例如系统调用返回原进程）

#### 3. 用户级线程 vs 内核级线程
| 特性 | 用户级线程 | 内核级线程 |
|------|-----------|-----------|
| **线程管理** | 用户空间的线程库 | 操作系统内核 |
| **内核感知** | 内核不知道线程存在 | 内核完全感知 |
| **阻塞行为** | 一个线程阻塞，整个进程阻塞 | 一个线程阻塞，其他线程可继续运行 |
| **多处理器** | 不能同时运行在多个CPU上 | 可以并行运行在多个CPU上 |
| **切换开销** | 非常小，无需陷入内核 | 较大，需要进程上下文切换 |

> **多对多模型**：多个用户级线程映射到多个内核级线程 → 既保留灵活性，又避免单线程阻塞问题

#### 4. 自旋锁（Spinlock）
- **工作原理**：线程不进入阻塞，而是循环检查锁是否可用
- **适用场景**：临界区执行时间**很短**的场景
- **不适用场景**：临界区执行时间长 → 浪费CPU
- **优点**：避免上下文切换开销
- **缺点**：忙等待浪费CPU，可能导致优先级反转

#### 5. PV操作（信号量）
- **P操作（wait）**：申请资源，s.count--，如果s.count < 0则加入等待队列s.queue并阻塞
- **V操作（signal）**：释放资源，s.count++，如果s.count ≤ 0则从s.queue唤醒一个等待进程
- **原子性**：PV操作必须是原子操作（不可中断）
- **注意**：P操作顺序会影响信号量最终值吗？**不会**！S = S - 1 - 1，结果与顺序无关

#### 6. Peterson算法（经典互斥算法）
```c
void enter_region(int process) {
    int other = 1 - process;
    interested[process] = TRUE;   // 表示我想进入
    turn = process;               // 主动让优先级给对方
    while (turn == process && interested[other] == TRUE);  // 等待对方不需要或谦让给我
}

void leave_region(int process) {
    interested[process] = FALSE;  // 表示我已离开临界区
}
```

#### 7. Dekker算法（最早的正确互斥算法）
核心思想：设置标志位表示想进入，同时设置turn变量表示"轮到谁"，通过主动谦让避免死锁

#### 8. 经典同步问题：读者写者（变种）
2025年考题：公园售票系统
- **查询者Q**：读操作，允许多个Q并发读
- **购票者B**：写操作，只能互斥访问
- **额外约束**：按到达顺序访问，余票为0时不允许写
- **解法要点**：使用互斥信号量保护写，使用计数器控制读者并发，保证读写公平

## 六、易错点提醒

1. **覆盖是在同一程序内部进行的**，不是不同作业之间
2. **增加可运行线程数量不能解决抖动**，反而可能加剧
3. **多级页表级数越多，访存性能越差**（因为需要多次查页表）
4. **fork后父子进程同一变量虚拟地址相同，但物理地址不同**（写时复制前可能共享物理页）
5. **线程虽然共享地址空间，但必须有自己独立的栈**
6. **用户级线程不能修改自己的页表项**（页表是内核管理的）
7. **两个不同进程的页表项可能内容相同**（共享内存时）
8. **虚拟地址空间不一定大于物理地址空间**（取决于具体实现）
