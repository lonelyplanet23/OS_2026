# 进程同步与互斥

---

## 一、并发的基础概念与制约关系

### 1. 并发引发的核心问题

| 术语 | 定义 |
|------|------|
| **竞争 (Race)** | 多个进程并发访问同一共享数据，结果不可预测，取决于相对执行次序 |
| **竞争条件 (Race Condition)** | 并发访问共享数据，执行结果与访问的特定顺序有关（Bernstein条件） |
| **临界资源** | 一次仅允许**一个进程**访问的资源（如打印机、共享变量） |
| **临界区 (Critical Section)** | 每个进程中**访问临界资源的那段代码** |

### 2. 两种基本制约关系

- **进程互斥（间接制约关系）**：**多个进程****不能同时进入**关于同一组共享变量的**临界区域**，是程序不希望的竞争关系
- **进程同步（直接制约关系）**：为完成同一任务的进程之间，因需要协调工作而**相互等待、交换信息**产生的刻意安排的关系（如：A执行完，B才能执行）

---

## 二、临界区管理准则

### 1. 理论条件（正确、公平、效率）

1. 没有进程在临界区时，想进入的进程可进入
2. **任何两个进程都不能同时进入临界区 (Mutual Exclusion)** ✨【核心】
3. 临界区外面的进程不能妨碍其他进程进入 (Progress)
4. 任何进入临界区的要求应在有限时间内得到满足 (Bounded Waiting)

### 2. 机制设计四大准则

| 准则 | 含义 |
|------|------|
| **空闲让进** | 资源空闲，允许进程进入 |
| **忙则等待** | 已有进程在执行，其他进程必须等待 |
| **有限等待** | 保证有限时间内能进入，避免死等 |   
| **让权等待** | 不能进入临界区时，应**立即释放CPU**，避免忙等空耗 |

## 三、基于忙等待的互斥方法

**共同缺陷**：违背"让权等待"准则，产生**忙等 (Busy Waiting)**，浪费CPU时间，且可能导致**优先级反转**。

### 1. 软件方法演进

| 方法 | 核心思想 | 问题 |
|------|----------|------|
| 尝试1 | 先检查标志再设置 | 检查与设置之间有空隙，无法互斥 |
| 尝试2 | 强制轮流（turn变量） | 严格交替，临界区外进程阻止其他进程，违反Progress |
| 尝试3 | After you（双方都设true） | 都想进入时可能都无法进入，死锁 |
| **Dekker算法** | 标志位 + turn变量，竞争时主动让权 | 第一个正确的软件算法，仍有忙等 |
| **Peterson算法** | interested数组 + turn，更简洁 | 完美解决互斥，考试常考补全 |
| **Lamport面包店算法** | 取号排队，号码小先服务 | N进程，彻底解决饥饿 |

### dekker 忙等待互斥算法
```
while(对方想进) {
    if(轮到对方) {
        我放弃;
        等对方用完;
        我重新申请;
    }
}
```

### 2. Peterson算法（必须掌握）

```c
#define FALSE 0
#define TRUE 1
#define N 2

int turn;                   // 轮到谁
int interested[N];          // 兴趣数组，初值均为FALSE

void enter_region(int process) {
    int other = 1 - process;
    interested[process] = TRUE;   // 表示本进程想进入
    turn = process;               // ✨【关键】主动把优先级让给对方
    while(turn == process && interested[other] == TRUE);  // 循环等待
}

void leave_region(int process) {
    interested[process] = FALSE;  // 表示已离开临界区
}
```

### 3. 硬件方法

- **中断屏蔽**：关中断进入临界区，开中断退出。简单高效，但不适用多处理器和用户进程
```
...
关中断;
临界区;
开中断;
```

- **Test and Set (TS / TSL)**：原语，真值表示已占用，false表示空闲，读出旧值并设置为True。MIPS通过LL（加载链接）和SC（条件存储）实现自旋锁
```C++
while TS(&lock); //加锁
临界区代码段;
lock = false; //上锁
其它代码;
```

- **Swap / XCHG**：原子地交换两个字的内容
```C++
    bool key = true;
    while(key != false)
    {
        Swap(&key, &lock); //循环
    }
    临界区代码段;
    lock = false; //上锁
    其它代码;
```

### 4. 互斥锁（自旋锁）✨【考试重点】

#### 核心概念
+ **本质**：基于TS指令实现的忙等锁
+ **基本思想**：获取不到锁时，**不放弃CPU**，循环空转（"自旋"）等待锁释放
+ **适用场景**：临界区非常短的情况（如信号量P/V操作本身）

#### 自旋锁实现互斥
```c
boolean lock = false;  // 共享锁变量，初始为false

// 进入临界区
while (TestAndSet(&lock));  // 忙等：旧值为true则一直循环
/* 临界区代码 */

// 退出临界区
lock = false;  // 释放锁
```

#### 特点分析
| 优点 | 缺点 |
|------|------|
| 无需上下文切换，**短临界区**效率高 | **忙等**，长临界区浪费CPU |
| 实现简单 | 优先级反转问题 |
| 多核场景下常用 | 单核CPU忙等无意义（占用CPU导致锁无法释放） |

## 四、信号量机制（Semaphore）✨【全章核心】

由Dijkstra提出，通过 `Sleep` 和 `Wakeup` 原语将"忙等"变为"阻塞"，完美解决忙等问题。

### 1. 信号量的定义与物理意义

```c
typedef struct semaphore {
    int count;           // 信号量值
    struct process *q;   // 等待队列（进程被阻塞）
} semaphore;
```

| S.count 值 | 物理意义 |
|------------|----------|
| **S > 0** | 表示当前**可用资源的数量**【临界区允许访问的进程数量】 |
| **S = 0** | 表示当前没有空闲资源 |
| **S < 0** | 绝对值 `|S|` 表示当前**被阻塞（等待该资源）的进程个数** |

### 2. P、V操作（必须成对且原子）

**P操作 (wait / down)**：申请资源
```c
void P(semaphore s) {
    s.count--;              // 先减1！不管是否阻塞
    if(s.count < 0)
        block(s.q);         // 阻塞，加入等待队列并重新调度
}
```

**V操作 (signal / up)**：释放资源
```c
void V(semaphore s) {
    s.count++;              // 先加1！不管是否唤醒
    if(s.count <= 0)
        wakeup(s.q);        // 从队列唤醒一个等待进程变为就绪态
}
```

> ✨【选择题高频考点】进程A、B调用P(S)各一次后，信号量S的值与调用顺序有关？
> ❌ 错误！S = S - 1 - 1，结果与顺序无关

### 3. 信号量的典型应用

| 应用场景 | 初值设置 | 使用方法 |
|----------|----------|----------|
| **互斥 (Mutex)** | 初值 = **1** | `P(S)` 和 `V(S)` 夹住**临界区** |
| **同步 (Sync)** | 初值 = **0** | 前置任务完成后 `V(S)`，后置任务开始前 `P(S)` |
| **汇合 (Rendezvous)** | 2个初值=0 | 交叉同步：A1在B2前，B1在A2前 |
| **屏障 (Barriers)** | 多个进程 | 最后一个到达的唤醒所有进程 |

## 五、经典同步与互斥问题 ✨

### 1. 生产者-消费者问题

**信号量设置**：
- `mutex = 1`：保护缓冲区互斥访问
- `empty = N`：空缓冲区数目
- `full = 0`：满缓冲区数目

```c
生产者:                    消费者:
while(true) {             while(true) {
    生产产品;                 P(full);   // 申请满缓冲
    P(empty);  // ✨申请空缓冲   P(mutex);  // ✨再申请进入
    P(mutex);  // ✨再申请进入    从缓冲区取产品;
    放入缓冲区;                V(mutex);
    V(mutex);                 V(empty);
    V(full);                  消费产品;
}                         }
```

⚠️ **死锁陷阱**：**必须先 P(empty)，再 P(mutex)**！
- 如果颠倒：先拿互斥锁，再等空位，当缓冲区满时，生产者拿着互斥锁去等空位，消费者想拿互斥锁取产品 → **死锁**
- V操作顺序随意，互换不会引发死锁

### 2. 读者-写者问题

**与消费者区别：** 写者完全不会影响文件本身，可以多个。

**规则**：读读允许；读写互斥；写写互斥

**信号量设置**：
- `rmutex = 1`：保护readcount计数器
- `wmutex = 1`：保护写操作互斥
- `readcount = 0`：读者计数

```c
读者:                      写者:
P(rmutex);                 P(wmutex);
readcount++;               写操作;
if(readcount == 1)         V(wmutex);
    P(wmutex);  // 第一个读者锁住数据，阻止写者
V(rmutex);

读操作;

P(rmutex);
readcount--;
if(readcount == 0)
    V(wmutex);  // 最后一个离开的读者释放数据
V(rmutex);
```

**特点**：基础算法是**读者优先**，只要有读者在读，后续读者可源源不断进入，可能导致**写者饿死**

+ **什么是优先：** 队伍中A在B前，但B能够插入到A之前先执行，那么B优先。

**读写公平**
```c
读者:                      写者:
P(enter);                  P(enter); 
P(rmutex);                 P(wmutex);
readcount++;               写操作;
if(readcount == 1)         V(wmutex);
    P(wmutex);             V(enter);
     // 第一个读者锁住数据，阻止写者
V(rmutex);
V(enter);

读操作;

P(rmutex);
readcount--;
if(readcount == 0)
    V(wmutex);  // 最后一个离开的读者释放数据
V(rmutex);
```
**根本区别：** 读者优先只有第一名读者和写者**竞争wmutex锁**，后续多个读者可以直接插队进入。而读写公平的版本下，每一个读者都要竞争enter这把锁。

【】请读者思考 如果互换了读者 `P(enter)`与`P(rmutex)`是否会产生死锁
+ 会，但是链条很长，例子如下：
```
读者1读完 → 需要拿 rmutex 来做 readcount--
    ↓ 等待
读者2持有 rmutex → 等待 enter
    ↓ 等待
写者W持有 enter → 等待 wmutex
    ↓ 等待
读者1正在读，持有 wmutex
    ↓ 循环
回到开头
```

### 3. 哲学家进餐问题

**问题**：5人5根筷子，每人需要2根。都先拿左边会导致死锁

**破局思路（考试常见解答）**：
1. 破坏请求与保持：最多允许4人同时入座（设置 `room = 4`）
```C
semaphore array[0..4] fork ← [1, 1, 1, 1, 1]

   semaphore room ← 4
   loop forever
       p1 : think                
       p2 : wait(room)                        
       p3 : wait(fork[i])                                  
       p4 : wait(fork[i + 1])                                           
       p5 : eat                                                
       p6 : signal(fork[i])         
       p7 : signal(fork[i + 1])                  
       p8 : signal(room)
```
(from geeksforgeeks.org)
2. 破坏循环等待：奇数号先拿左筷子再拿右筷子，偶数号相反
3. 同时申请：使用AND型信号量，同时拿起两根，否则不拿

### 大题格式


```c
Semaphore full = 0;
Semaphore empty = n;
Semaphore mutex = 1;
ItemType buffer[0..n-1];
int in = 0, out = 0;
```
**main**
```c
main(){
    Cobegin
    producer();
    Consumer();
    Coend
}
```
**Producer**：
```c
producer() {
    while(true) {
        生产产品nextp;
        P(empty);
        P(mutex);
        buffer[in] = nextp;
        in = (in + 1) MOD n;
        V(mutex);
        V(full);
    }
}
```

**Consumer**：
```c
consumer() {
    while(true) {
        P(full);
        P(mutex);
        nextc = buffer[out];
        out = (out + 1) MOD n;
        V(mutex);
        V(empty);
        消费nextc中的产品
    }
}
```
+ 共享变量；一定需要用互斥锁夹住。


### 理发师问题

**问题描述**：理发店里有1位理发师、1把理发椅和n把供等候理发的顾客坐的椅子；

- 如果没有顾客，理发师便在理发椅上睡觉，当一个顾客到来时，叫醒理发师；
- 如果理发师正在理发时，又有顾客来到，则如果有空椅子可坐，就坐下来等待，否则就离开。

**互斥资源**：理发师、顾客、椅子

**同步约束**：访问椅子、顾客唤醒理发师、理发师唤醒下一位等待顾客

```c
int waiting = 0;  // 等待顾客数
Semaphore mutex = 1;
Semaphore customer = 0, barber = 0; //同步
Barber:
while(true) {
    P(customer); //接待顾客
    
    P(mutex);
    waiting --;
    V(mutex);

    V(barber); //唤醒顾客
    CutHair();
}

Custom_i:
P(mutex)
if (waiting < N) {
    waiting++;
    V(mutex);   //防止死锁，mutex要锁到最小的访问waiting的区域。
    V(customer) //唤醒理发师

    P(barber) //等待并接受理发
    GetHairCut();
}
else
{
    离开;
    V(mutex);
}
```

## 六、管程与条件变量 ✨【选择/简答考点】

### 1. 为什么需要管程？

信号量的PV操作**分散在各个进程中**，容易因写错顺序导致死锁（如先P(mutex)再P(empty)）。

**管程核心思想**：将共享变量和操作**封装集中**在一起，由编译器保证互斥。

### 2. 管程的定义与特性

**定义**：一个由过程、变量及数据结构组成的集合，封装了同步互斥逻辑。

**三大组成**：
| 组成 | 作用 |
|------|------|
| **共享数据** | 被保护的临界资源 |
| **操作过程** | 操作临界资源的函数 |
| **条件变量** | 不同原因被阻塞的队列 + wait/signal操作 |

**核心特性**：
1. ✅ **互斥性**：管程天然自带互斥锁，**任何时刻只能有一个进程在管程内活动**
2. ✅ **封装性**：同步互斥逻辑封装在管程内部，外部不可见
3. ✅ **安全性**：编译器自动处理互斥，减少程序员出错机会

> 📌 管程是**语言级的机制**，由编译器负责实现互斥，不是操作系统直接提供的机制。

### 3. 条件变量（Condition Variable）

信号量有"值"的概念，但条件变量**没有值**，每一个条件变量保存了一个**等待队列**

| 操作 | 作用 |
|------|------|
| **wait(cond)** | 等待该条件满足，调用时**自动释放管程锁**，被唤醒时重新获得锁 |
| **signal(cond)** | 唤醒一个在该条件上等待的进程 |

> ⚠️ **关键理解**：调用wait时必须释放管程锁，否则其他进程永远无法进入管程，条件永远无法满足，导致死锁！

### 4. 信号量 vs 条件变量

| 特性 | 条件变量 | 信号量 |
|------|----------|--------|
| **有无值** | 没有值，不可增减 | 有整型值 |
| **信号丢失** | 没有等待进程时，signal信号**直接丢失** | V操作会使值+1，不会丢失 |
| **wait行为** | wait会立即阻塞并**释放管程的互斥锁** | P操作阻塞但不释放其他锁 |

### 5. 管程实现生产者-消费者问题

```c
monitor ProducerConsumer
    condition full, empty;  // 条件变量，不是信号量！不含值
    integer count;

    procedure insert(item: integer);
    begin
        if count == N then wait(full);
        insert_item(item);
        count++;
        if count == 1 then signal(empty);
    end;

    function remove: integer;
    begin
        if count == 0 then wait(empty);
        remove = remove_item;
        count--;
        if count == N-1 then signal(full);
    end;

    count := 0;
end monitor;
```

## 七、进程通信（IPC）✨【简答题考点】

### 1. 进程通信的定义

**进程通信（Inter-Process Communication, IPC）**：指进程之间的信息交换。

### 2. 五种主要通信类型

| 类型 | 核心思想 | 特点 | 典型实现 |
|------|---------|------|---------|
| **共享存储器** | 进程共享内存区域直接读写 | 速度最快，需自行处理同步 | 共享内存 |
| **消息传递** | 通过OS提供的send/receive原语 | OS负责同步，可跨网络 | 消息队列、邮箱 |
| **管道通信** | 以文件为中间介质，连接读写进程 | 流式传输，简单易用 | 匿名/命名管道 |
| **套接字Socket** | 跨网络的标准化接口 | 支持本机和网络通信 | TCP/UDP Socket |
| **信号Signal** | 软中断机制，通知事件发生 | 只传递控制信息，不传递数据 | 信号 |

### 3. 共享存储器系统

**两个阶段**：
1. **建立共享存储区**：一个进程请求创建，其他进程附接到自己地址空间
2. **读写操作**：像访问普通内存一样读写

**特点**：
- ✅ 速度最快，无中间拷贝
- ❌ 需要自己处理同步互斥
- ❌ 只适用于同一台机器

### 4. 消息传递系统

**两种实现方式**：
| 方式 | 特点 | 适用场景 |
|------|------|---------|
| **直接通信** | 发送者直接指定接收者，消息挂在接收进程的消息队列 | 一对一通信 |
| **间接通信** | 消息发送到中间信箱，接收者从信箱取信 | 多对多、广播 |

## 八、考试重点总结 & 易错点

### 🎯 必考点（按考察概率排序）

1. **Peterson算法填空**：`turn = process` 是关键！主动谦让给对方
2. **PV操作的物理意义**：S > 0 可用资源数；S < 0 等待进程数
3. **生产者-消费者P操作顺序**：同步信号量在前，互斥信号量在后，否则死锁
4. **读者-写者变种设计**：识别读者写者，设计计数器和信号量
5. **信号量与条件变量的区别**：信号有记忆，条件变量无记忆
6. **自旋锁的特点与适用场景**：短临界区用，长临界区不用

### ⚠️ 判断题易错点

| 陈述 | 判断 | 说明 |
|------|------|------|
| 信号量P操作阻塞时，不会占用CPU资源 | ✅ 正确 | 阻塞了就释放CPU了 |
| 使用信号量既可以实现互斥，也可以实现同步 | ✅ 正确 | 初值不同而已 |
| 自旋锁临界区时间越长越高效 | ❌ 错误 | 临界区越短越高效 |
| 两个进程先后执行P(S)，结果与顺序有关 | ❌ 错误 | 都是S-2 |
| 互斥是同步的一种特殊情况 | ✅ 正确 | 同步包含互斥 |
| 管程中wait条件变量时，不需要释放锁 | ❌ 错误 | 必须释放，否则死锁 |
| 条件变量有值，可以像信号量一样加减 | ❌ 错误 | 条件变量只是等待队列标识，没有值 |
| 管程允许多个进程同时在管程内活动 | ❌ 错误 | 任何时刻只能有一个进程在管程内 |
