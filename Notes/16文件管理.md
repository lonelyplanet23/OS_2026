### 📚 一、文件基本概念：为什么要引入“文件”？

#### 1. 数据持久化存储的需求

计算机中的程序运行时可以把数据放在进程自己的地址空间中，但这种方式有三个明显问题：

| 需求 | 含义 | 仅靠进程地址空间的问题 |
| --- | --- | --- |
| 存储大量数据 | 数据规模可能远超内存或进程地址空间 | 用户地址空间有限 |
| 长期保存 | 进程终止后数据仍应存在 | 进程结束后地址空间消失 |
| 可以共享数据 | 多个进程、多个用户可能访问同一份数据 | 进程私有地址空间不便共享 |

因此，操作系统提出“文件”概念，把数据组织成文件形式，用文件作为数据的存储和访问单位。

**提出“文件”概念**，作为数据储存与访问单位。【抽象提炼】

为什么不用扇区、页直接作为用户访问单位？因为文件更能满足“大量存储、长期保存、共享访问”这三个基本要求，并且能够屏蔽具体存储介质和磁盘块布局细节。

```text
用户需要保存数据
      │
      ├─ 数据量大：超过进程地址空间
      ├─ 生命周期长：进程结束后仍存在
      └─ 需要共享：多个进程/用户访问
      │
      ▼
操作系统抽象出“文件”
      │
      └─ 用符号名 + 字节序列组织持久化数据
```

#### 2. 文件的定义

文件是具有符号名、由字节序列构成的数据项集合，是文件系统的基本数据单位。

**概念: 一组带标识的、在逻辑上有完整意义的信息项的序列**

可以从几个角度理解文件：

| 角度 | 理解 |
| --- | --- |
| 用户视角 | 带名字的、逻辑上完整的数据集合 |
| 操作系统视角 | 若干数据块及其元数据的组合 |
| 地址空间视角 | 一个单独连续的逻辑地址空间，大小即文件大小 |
| 抽象视角 | 屏蔽磁盘、扇区、块等实现细节的数据抽象机制 |

文件具有以下特征：

- 一个数据抽象机制；
- 一个单独连续的逻辑地址空间；
- 与进程的地址空间无关；
- 本质上是一组字节序列。

**信息项**：构成文件内容的**基本单位**，各信息项有顺序关系。

#### 3. 文件的构成：文件体与文件说明

文件通常包括两部分：

| 组成 | 含义 | 例子 |
| --- | --- | --- |
| 文件体 data | 文件本身的内容 | 文本内容、程序代码、图片数据 |
| 文件说明 metadata | 文件存储和管理相关信息 | 文件名、位置、大小、权限、时间戳 |

文件的构成：文件体(data)、文件说明（如：补充） (meta data)

文件说明也称为文件属性或元数据，是操作系统管理文件的关键依据。

#### 4. “一切皆文件”

在 UNIX / Linux 类操作系统中，普通文件、目录、设备、管道、套接字等都可以抽象为文件。

- linux 外设的访问也一切皆文件

| 类型 | 例子 | 说明 |
| --- | --- | --- |
| 普通文件 | `.txt`、`.c`、可执行文件 | 保存用户数据或程序 |
| 目录文件 | `/home`、`/usr` | 保存目录项，用于组织文件 |
| 字符设备文件 | 键盘、终端、打印机 | 按字符流访问 |
| 块设备文件 | 磁盘、光驱 | 按块访问 |
| 网络文件 | socket | 统一为文件式接口 |

这种抽象的好处是：所有 I/O 对象都可以通过类似 `open`、`read`、`write`、`close` 的统一接口访问。

### 🏗️ 二、文件管理与文件系统：操作系统如何管理文件？

#### 1. 文件管理的内容

**文件由操作系统管理：** 结构、命名、存取、使用、保护和实现方法。

文件管理需要解决：

1. 文件如何命名；
2. 文件如何组织成目录；
3. 文件如何读写和定位；
4. 文件如何映射到磁盘块；
5. 文件如何共享与保护；
6. 文件系统如何保证可靠性和性能。

#### 2. 文件系统的定义

文件系统是存储设备上组织文件的方法和数据结构，也是操作系统中负责文件命名、存储、检索、更新、共享与保护的子系统。

**文件系统**

- 有效的分配文件存储器的空间；
- 对用户透明的机制（具体的实现对用户不可见，只需要最基本的接口）。

文件系统可以理解为把“磁盘块”抽象成“用户能理解的文件名和文件内容”的中间层。

```text
用户看到：/tmp/test/hello.c
      │
      ▼
文件系统解析路径、查目录、查 inode/FCB
      │
      ▼
找到文件对应的逻辑块 → 物理块映射
      │
      ▼
磁盘驱动按块号访问磁盘
```

#### 3. 文件系统的任务

| 任务 | 内容 |
| --- | --- |
| 空间管理 | 统一管理磁盘空间，分配与回收空闲块 |
| 按名存取 | 把名字空间映射到磁盘空间 |
| 共享与保护 | 支持共享，提供权限、保密和安全机制 |
| 用户接口 | 提供创建、删除、读写、打开、关闭等操作 |
| 性能优化 | 减少磁盘访问次数，提高读写效率 |
| I/O 统一接口 | 与设备管理配合，为不同设备提供统一抽象 |

#### 4. 文件系统的层次

**文件系统的层次**

- 文件系统的接口（接口）；
- 对象 OS 的软件集合（行为）；
  - 存储空间管理、目录管理、用户将文件 VA 转化为 PA；
  - 读写管理；
  - 共享保护；
- 对象及属性（数据结构）；
  - 文件；
  - 目录；
  - 磁盘储存空间。

可以整理为：

```text
文件系统层次
  │
  ├─ 用户接口层
  │   └─ create / delete / open / close / read / write / seek
  │
  ├─ 文件管理软件层
  │   ├─ 目录管理
  │   ├─ 存储空间管理
  │   ├─ 文件读写管理
  │   └─ 共享与保护
  │
  └─ 基本数据结构层
      ├─ 文件
      ├─ 目录
      ├─ FCB / inode
      └─ 磁盘块 / 空闲空间
```

### 📁 三、用户视角的文件：名称、类型、访问方式与基本操作

#### 1. 文件名

**文件名** 有限长度的字符串  
**文件名.扩展名**  扩展名：提示操作系统如何打开解析文件

例如：

```text
hello.c
report.docx
main.exe
```

注意：扩展名不一定决定文件本质内容，它更多是给操作系统、应用程序和用户提供解释文件格式的提示。

#### 2. 文件类型

文件类型可以按不同标准分类：

| 分类标准 | 类型 |
| --- | --- |
| 按性质和用途 | 系统文件、库文件、用户文件 |
| 按数据形式 | 源文件、目标文件、可执行文件 |
| 按保护级别 | 只读文件、读写文件、执行文件、不保护文件 |
| 按逻辑结构 | 有结构文件、无结构文件 |
| 按物理结构 | 顺序文件、链接文件、索引文件 |

**文件类型**

- 按文件中物理结构分：顺序文件、链接文件、索引文件；
- 按数据形式：源文件、目标文件、可执行文件。

#### 3. 文件逻辑结构

文件逻辑结构是从用户角度看文件，由用户的访问方式确定。

| 逻辑结构 | 含义 | 典型例子 |
| --- | --- | --- |
| 流式文件 | 文件是无结构的字节序列 | UNIX 普通文件 |
| 记录式文件 | 文件由若干记录组成，可按记录访问 | 数据库记录、表格记录 |
| 树形结构 | 文件内容组织成树 | 某些索引式数据结构 |

#### 4. 文件存取方式

**文件存取方式**包括：

| 方式 | 含义 | 适合场景 |
| --- | --- | --- |
| 顺序访问 | 从前到后依次读写 | 文本扫描、日志读取 |
| 随机访问 | 指定当前位置，从任意位置读写 | 数据库、数组/矩阵文件 |
| 索引访问 | 根据关键字或索引查找 | 数据库索引 |

顺序访问类似“磁带式”读取；随机访问则要求文件系统能够快速把逻辑偏移转换为物理块位置。

#### 5. 文件基本操作

**文件基本操作**包括：

| 操作 | 含义 |
| --- | --- |
| Create | 创建文件 |
| Delete | 删除文件 |
| Open | 打开文件，获得文件描述符 |
| Close | 关闭文件 |
| Read | 读文件 |
| Write | 写文件 |
| Append | 追加写 |
| Seek | 设置文件读写位置 |
| Get attributes | 获取文件属性 |
| Set attributes | 设置文件属性 |
| Rename | 修改文件名 |

应用访问文件数据前必须先打开文件，获得文件描述符：

```c
fd = open(name, flag);
read(fd, ...);
close(fd);
```

文件描述符形式上是非负整数，实际上是一个索引值，指向内核为每个进程维护的打开文件表。

### 🧭 四、目录与路径：如何通过名字找到文件？

#### 1. 目录的定义

### 目录

目录：由**文件说明索引**组成的用于文件检索的**特殊文件**

- 是文件；
- 内容是由指针组成。

更准确地说，目录是一类特殊文件，它的内容主要是目录项。目录项通常记录：

```text
<文件名, 指向文件控制块/索引节点的指针>
```

或在某些系统中直接包含 FCB。

**典型的文件系统组织**：Dir + files

#### 2. 目录需要解决的问题

当文件数量很多时，目录需要解决：

| 问题 | 说明 |
| --- | --- |
| 效率 | 能迅速定位一个文件 |
| 命名 | 允许不同目录下出现相同文件名 |
| 别名 | 同一个文件可以有多个名字 |
| 分组 | 把文件按课程、用户、类型等分组 |
| 共享 | 多个用户访问同一份文件 |

别名：同一个文件可以有多个别名。  
**目录内容：文件名，别名的数目**  
**文件类型、地址信息、访问控制信息、使用信息**

#### 3. 目录结构的演化

##### 单级目录

所有文件都放在同一个目录中。

| 优点 | 缺点 |
| --- | --- |
| 结构简单，实现容易 | 文件多时检索慢 |
| 早期个人计算机可用 | 容易命名冲突 |
|  | 不便于共享与分组 |

课堂随记中写作“单击文件目录”，这里应整理为“单级文件目录”。

##### 两级目录

第一级是根目录，第二级是用户目录。每个用户有自己的目录。

| 优点 | 缺点 |
| --- | --- |
| 不同用户可有同名文件 | 用户目录下不能继续分层 |
| 比单级目录更适合多用户系统 | 对复杂文件组织仍不够方便 |

##### 多级目录（层次目录）

现代文件系统几乎都采用多级目录结构。

**两级目录、多级目录**：最后一级目录，指向文件的物理地址

更准确地说，在基于 inode 的文件系统中，目录项一般指向 inode / FCB，而不是直接保存文件数据块地址。

```text
多级目录
/
├─ home
│  └─ user
│     └─ note.md
└─ tmp
   └─ test
      └─ helloworld.c
```

多级目录的特点：

| 特点 | 说明 |
| --- | --- |
| 层次清楚 | 不同性质、不同用户文件可构成不同子树 |
| 解决重名 | 只要求同一目录下文件名不重复 |
| 查找较快 | 每次只查找目录树中的一个子集 |
| 便于保护 | 不同目录可设置不同权限 |
| 代价 | 目录级别太多时路径检索时间增加 |

#### 4. 路径

#### 路径 - 描述一个文件的位置

“你很熟悉的”

| 路径类型 | 含义 | 例子 |
| --- | --- | --- |
| 绝对路径 | 从根目录开始描述 | `/tmp/test/helloworld.c` |
| 相对路径 | 从当前目录开始描述 | `../notes/16文件管理.md` |
| 当前目录 | `.` | 当前所在目录 |
| 上一级目录 | `..` | 父目录 |

**绝对路径：** 从根目录 `/` 开始。  
**相对路径：** 从当前工作目录开始。  
**上一级目录**：通常用 `..` 表示。

### 🧱 五、文件系统实现：FCB、inode 与逻辑块到物理块的映射

#### 1. 根本问题：逻辑对象到物理对象的映射

### 文件系统实现

### 根本问题：逻辑对象到物理对象的映射

用户看到的是文件名和文件内偏移；磁盘只能按物理块读写。因此文件系统必须完成：

```text
文件名 / 路径
      │
      ▼
目录项查找
      │
      ▼
FCB / inode
      │
      ▼
逻辑块号 → 物理块号
      │
      ▼
磁盘块读写
```

文件实现需要解决两个问题：

1. 如何描述一个文件，记录文件各种管理信息？
2. 如何把文件的连续逻辑块存放到磁盘空闲物理块中，并记录逻辑块到物理块的映射关系？

#### 2. FCB 文件控制块

### FCB 文件控制块

文件控制块（FCB, File Control Block）是为管理文件而设置的数据结构，保存管理文件所需的所有有关信息。

**基本信息：**

| 信息 | 说明 |
| --- | --- |
| 文件名 | 文件的符号名 |
| 文件号 | 文件内部标识 |
| 文件大小 | 文件字节数或块数 |
| 物理位置 | 文件数据块位置或索引表位置 |
| 文件逻辑结构 | 流式、记录式等 |
| 文件物理结构 | 连续、链接、索引等 |

**访问控制信息：**

| 信息 | 说明 |
| --- | --- |
| 文件所有者 | 创建者或当前拥有者 |
| 访问权限 | 读、写、执行、删除等 |
| 共享计数 | 被引用或共享情况 |

**使用信息：**

- 创建时间，上一次修改时间；
- 最后访问时间；
- 当前使用信息；
- 只读、隐藏、系统、归档等标志。

#### 3. inode 的思想

Unix 类文件系统常用 inode 描述文件。inode 保存文件或目录的属性信息和数据块指针，但通常不直接保存文件名。文件名保存在目录项中。

| 结构 | 保存内容 |
| --- | --- |
| 目录项 | 文件名 + inode 号 / inode 指针 |
| inode | 文件类型、大小、权限、时间戳、数据块指针 |
| 数据块 | 文件实际内容或目录项内容 |

```text
目录项：hello.txt → inode #10
                    │
                    ▼
inode #10：权限、大小、时间、数据块指针
                    │
                    ▼
数据块：hello.txt 的真实内容
```

💡 **期末重点：** 目录项中一般不保存文件数据所在物理块指针，而保存索引节点所在位置；inode 再保存文件元数据和数据块指针。

### 🧩 六、文件物理结构：连续、链接与索引

### 文件逻辑结构 物理结构

文件物理结构主要解决：

- 假设一个文件划分为 N 块，这 N 块在磁盘上如何存放？
- 地址在 FCB 如何记录的？

常见结构包括连续结构、串联/链接结构、索引结构。

#### 1. 连续（顺序）结构 -- 顺序表

#### 连续（顺序）结构 -- 顺序表

一维的条带结构：起始地址+长度

连续结构把一个文件的信息存放在若干连续的物理块中。FCB 中只需记录起始块号和文件长度。

![alt text](assets/image-44.png)

| 项目 | 内容 |
| --- | --- |
| 记录方式 | 起始块号 + 块数 / 长度 |
| 逻辑块 i 的物理块 | 起始块号 + i |
| 适合 | 变化不大、顺序访问为主的文件 |

优点：还可以随机读取（逻辑偏移转换为物理偏移很简单），连续读取速度快

缺点：文件长度固定不易改变，不利于文件动态增加或修改

整理如下：

| 优点 | 缺点 |
| --- | --- |
| 结构简单，实现容易 | 文件长度一经固定不易改变 |
| 不需要额外指针空间 | 不利于动态增长和修改 |
| 支持顺序存取和随机存取 | 可能产生外部碎片 |
| 顺序存取速度快 | 创建时可能需要预估大小 |

#### 2. 串联/链接文件结构 -- 链表

#### 串联/链接文件结构 -- 链表

每个物理块的最后一个字作为链接字，指向后继块的物理地址

![43](assets/image-43.png)

链接结构中，文件由若干不一定连续的物理块组成，每个块中保存指向下一块的指针。FCB 中保存链首指针，文件结尾块指针为空或特殊标记。

| 项目 | 内容 |
| --- | --- |
| 记录方式 | FCB 保存首块地址，每块保存下一块地址 |
| 适合 | 动态增长、顺序访问较多的文件 |
| 结尾标记 | 空指针或特殊值 |

优点：空间利用率高，文件动态扩充修改容易，顺序存取效率高（链表）  
缺点：**随机存储效率太低**，链接指针占用一定空间

整理如下：

| 优点 | 缺点 |
| --- | --- |
| 空间利用率高 | 随机访问效率低，访问第 i 块需从头遍历 |
| 文件动态扩充容易 | 指针出错会影响可靠性 |
| 不要求连续空间 | 链接指针占用空间 |
| 适合顺序访问 | 指针分散导致缓存局部性较差 |

##### FAT：显示链接

显示链接 FAT file allocation table 【为磁盘所有盘块建立链接表】

文件分配表中：下标为盘块号，存放指向下一个盘快的指针，-1表示最后一块，-2表示空闲等等

更准确地说，FAT（File Allocation Table）为每个簇/块设置一个表项：

| FAT 表项值 | 含义 |
| --- | --- |
| 0 | 空闲 |
| 坏簇标记 | 物理坏块 |
| 下一簇号 | 当前簇被文件占用，且下一簇为该值 |
| EOF / FFFF | 文件最后一簇 |

FAT 的本质是把“每个块中的链接指针”集中放到内存中的文件分配表里，减少读文件时频繁读取块内指针的开销，但 FAT 表本身也会占用空间，并且损坏后影响较大。

#### 3. 索引结构【重点!】

#### 索引结构 【重点!】

**索引表**：为**每个文件**建立一个数据结构，将该文件的**所有物理块块号存放在该索引中**。

课堂随记中写“下标是物理块号”，这里应修正为：索引表的下标通常是**文件逻辑块号**，表项内容是对应的**物理块号**。

```text
索引表
逻辑块号  0   1   2   3   ...
          │   │   │   │
          ▼   ▼   ▼   ▼
物理块号  8  21   5  40   ...
```

**索引文件**由索引表和数据区组成。访问索引文件通常需要两步：

- 第一次读取索引表物理块，第二次读取文件对应物理块

即：

```text
访问文件逻辑块 i
      │
      ├─ 1. 查索引表：逻辑块 i → 物理块号 p
      │
      └─ 2. 读取物理块 p 中的数据
```

![45](assets/image-45.png)

优点：保留了链式结构的优点，又回避了其缺点

- 可以顺序存取，又能随机存取（为什么？索引**表**）；
- 满足动态文件扩容，插入删除要求。

缺点：提高系统开销：存取时间、内外存空间。

整理如下：

| 优点 | 缺点 |
| --- | --- |
| 既支持顺序访问，也支持随机访问 | 索引表需要额外空间 |
| 文件可动态增长 | 访问时可能先读索引块，增加 I/O |
| 外存空间利用率高 | 大文件需要多级索引，结构更复杂 |
| 避免链接结构随机访问差的问题 | 小文件也可能有索引开销 |

#### 4. 索引表组织方式

**组织方式：**

##### 直接索引

**直接索引**

inode / FCB 中的地址项直接保存数据块地址。

![alt text](assets/image-46.png)

缺点：文件存储大小受索引文件大小的影响，需要多级索引解决

##### 多级索引（间接索引）

**多级索引（间接索引）**

- 可存储文件大小提高

一级间接索引：地址项指向一个索引块，索引块中再存放数据块地址。  
二级间接索引：地址项指向一级索引块，一级索引块中的每项再指向二级索引块。  
三级间接索引依此类推。

##### 综合模式：直接 + 间接

**综合模式：直接+间接**

有些指针域是直接索引，有些指针域指向一个新的一次索引块，有的指向二级索引块

![47](assets/image-47.png)

综合模式的核心思想：

| 文件大小 | 使用方式 | 好处 |
| --- | --- | --- |
| 小文件 | 直接索引 | 访问快，少一次索引块 I/O |
| 中等文件 | 一级间接索引 | 扩展文件容量 |
| 大文件 | 二级/三级间接索引 | 支持很大文件 |

💡 **常考：** 直接索引 + 多级间接索引支持的最大文件大小计算。

#### 5. 三种文件物理结构对比【常考】

**【常考】三者文件逻辑结构，优缺点比较**

| 结构 | FCB 中记录 | 顺序访问 | 随机访问 | 动态增长 | 空间利用 | 主要缺点 |
| --- | --- | --- | --- | --- | --- | --- |
| 连续结构 | 起始块号 + 长度 | 很快 | 很快 | 差 | 可能产生外部碎片 | 文件大小不易改变 |
| 链接结构 | 首块指针 | 较好 | 很差 | 好 | 较高 | 随机访问必须沿链查找 |
| 索引结构 | 索引表地址 / inode 指针 | 好 | 好 | 好 | 较高 | 索引表带来空间和时间开销 |

【考试重点】若选择题说“索引文件适用于顺序访问，但不适合随机访问”，这是错误的。索引文件既适合顺序访问，也适合随机访问。

#### 2. 基于 inode 的文件查找过程

以读取 `/foo/bar` 为例，假设根目录 inode 已经读入内存：

```text
读取 /foo/bar
  │
  ├─ 1. 根据根目录 inode，得到根目录数据块地址
  │
  ├─ 2. 读取根目录内容，查找目录项 "foo"
  │
  ├─ 3. 得到 foo 的 inode 号 / inode 所在块地址
  │
  ├─ 4. 读取 foo 的 inode，得到 /foo 目录数据块地址
  │
  ├─ 5. 读取 /foo 目录内容，查找目录项 "bar"
  │
  ├─ 6. 得到 bar 的 inode 号 / inode 所在块地址
  │
  ├─ 7. 读取 bar 的 inode，得到文件数据块地址
  │
  └─ 8. 根据数据块指针读取 bar 的实际内容
```

这正是 2021、2024 期末文件系统大题的高频考法。

### 🔗 八、文件共享：硬链接与软链接

#### 1. 可供共享的目录组织

#### 可供共享的目录组织

**文件共享**：系统**保留**该文件的**一个副本**，共享该文件的用户可用**相同或者不同的文件名**访问它。

文件共享的目标是在磁盘中只保存一份文件内容，但允许多个目录项指向它。

#### 2. 硬链接

硬链接是多个文件名指向同一个物理文件 / inode 的链接关系。它们具有相同 inode 号，可以位于不同目录中。

| 特点 | 说明 |
| --- | --- |
| inode 相同 | 多个目录项指向同一 inode |
| 地位平等 | 原始文件名和硬链接没有本质区别 |
| 删除一个名字不影响其他名字 | 只有链接计数为 0 时才释放文件 |
| 通常不能跨文件系统 | inode 号只在同一文件系统内有效 |

```text
目录项 A：a.txt ─┐
                 ├─ inode #100 ── 数据块
目录项 B：b.txt ─┘
```

#### 3. 软链接

软链接（符号链接）是一个特殊文件，它保存目标文件或目录的路径名。

| 特点 | 说明 |
| --- | --- |
| 独立文件 | 有自己的 inode |
| 保存路径 | 内容是目标路径字符串 |
| 可跨文件系统 | 因为保存的是路径 |
| 可能悬空 | 目标被删除后软链接失效 |

```text
link.txt ──保存路径──> /path/to/Foo.txt ──> inode #2433
```

#### 4. 硬链接 vs 软链接

| 对比项 | 硬链接 | 软链接 |
| --- | --- | --- |
| 本质 | 多个目录项指向同一 inode | 一个文件保存另一个文件路径 |
| inode | 与目标相同 | 与目标不同 |
| 删除目标后 | 其他硬链接仍可访问 | 软链接可能失效 |
| 是否可跨文件系统 | 通常不可以 | 可以 |
| 是否可链接目录 | 通常限制较多 | 可以链接目录 |

### 🔄 九、磁盘空间管理：如何记录空闲块？【重点】

文件系统不仅要记录已分配给文件的块，还要管理哪些磁盘块是空闲的。

## 磁盘空间的管理 【重点2】

#### 1. 空闲表法

### 空闲表

记录第一个空闲盘块号、该空闲区的空号线盘块数

- 类似内存的动态分配

更准确地说，空闲表记录每段连续空闲区的起始块号和空闲块数。

| 起始块号 | 空闲块数 |
| ---: | ---: |
| 10 | 5 |
| 30 | 8 |
| 100 | 20 |

| 优点 | 缺点 |
| --- | --- |
| 适合连续分配 | 表项可能很多 |
| 查找连续空闲区方便 | 分配回收时需要合并相邻空闲区 |

#### 2. 位图法

### 位图法

位图法为每个磁盘块设置一位，用 0/1 表示空闲或已分配。

```text
块号： 0 1 2 3 4 5 6 7 ...
位图： 1 1 0 0 1 0 1 1 ...
```

| 优点 | 缺点 |
| --- | --- |
| 结构简单 | 位图本身占空间 |
| 容易查找连续空闲块 | 大磁盘扫描位图可能耗时 |
| 适合用位运算加速 | 需要常驻或频繁读入内存 |

#### 3. 空闲链表法

### 空闲链表法

**空闲盘块链：以盘块为基本元素成链**，每个盘块都有指向下一个空闲盘快的指针

**空闲盘区链：** **空闲盘区**内包含若干**连续空闲盘块**，每个盘区包含**本盘区的盘块数**，和**指向下一个空闲盘区的指针**（顺序和链表的结合）

| 类型 | 基本元素 | 特点 |
| --- | --- | --- |
| 空闲盘块链 | 单个空闲块 | 分配单块容易，但链很长 |
| 空闲盘区链 | 一段连续空闲块 | 兼顾连续空间信息，链较短 |

#### 4. 成组链接法【掌握】

### 成组链接法 【掌握】

**结构**

将空闲盘块分成**若干组**

**第一组**的空闲块信息记录在**超级块（专用块）**中

每组的**第一个空闲块**记录了**下一组空闲块的物理盘块号**，以及本组**空闲块总数**（为0表示结束）

其余的空闲块为**指向物理空闲块的指针**

![50](assets/image-50.png)

成组链接法可以理解为“栈 + 链接”的组合：超级块中保存一组空闲块号，分配时像栈一样弹出；当栈中剩下的特殊块被取出时，把该块中保存的下一组空闲块号读入超级块。

**分配过程**

**空闲盘块号栈: **
不断压入空闲盘块号栈，若压入的是栈底（自下而上），则将下一组盘块的内容压入栈（似乎描述不太准）

整理为更清晰的分配流程：

```text
成组链接法分配空闲块
  │
  ├─ 1. 超级块中维护一组空闲块号，类似栈
  │
  ├─ 2. 分配时从栈顶弹出一个空闲块号
  │
  ├─ 3. 若弹出的不是“下一组信息块”，直接分配给文件
  │
  └─ 4. 若弹出的是保存下一组信息的块，先把其中记录的下一组空闲块号读入超级块，再分配该块
```

回收流程则是把释放的块号压入超级块的空闲块号栈；若栈满，则把当前超级块中的一组空闲块号写入被回收块中，让它成为新的“下一组信息块”。

### 🚀 十一、文件系统性能：如何减少磁盘访问次数？

## 文件性能问题

**减少磁盘访问速度**

这里应理解为：减少磁盘访问次数、降低磁盘访问开销，或使用 SSD 等更快设备。

例如 SSD

**提高文件系统性能：...（所讲内容）**

#### 1. 文件系统性能瓶颈

磁盘服务速度是系统性能的主要瓶颈之一。机械磁盘访问中，寻道时间和旋转延迟往往远大于内存访问时间。

文件系统优化的核心：

```text
减少真实磁盘 I/O 次数
      │
      ├─ 缓存：命中则不访问磁盘
      ├─ 提前读：顺序访问时预取后续块
      ├─ 延迟写：合并多个写操作
      ├─ 优化布局：相关数据尽量靠近
      └─ 合理块大小：平衡吞吐与空间利用率
```

#### 2. 块高速缓存

**块高速缓冲** 替换算法

块高速缓存（Block Cache）是在内存中为磁盘块设置的缓冲区，保存磁盘中某些块的副本。

读请求过程：

```text
读请求到达
  │
  ├─ 检查块高速缓存
  │     ├─ 命中：直接从缓存读
  │     └─ 未命中：从磁盘读入缓存，再返回数据
  │
  └─ 根据局部性原理，后续可能再次访问该块
```

常见组织方式：双向链表 + Hash 表。

| 结构 | 作用 |
| --- | --- |
| 双向链表 | 管理缓存块替换顺序，如 LRU |
| Hash 表 | 根据块号快速判断是否在缓存中 |

#### 3. 磁盘块大小权衡

**磁盘块大小：块大，读取次数少但浪费空间；块小，利用率高，但需要多次读写**

| 块大小 | 优点 | 缺点 |
| --- | --- | --- |
| 较大 | 顺序读写效率高，索引项相对少 | 内部碎片大，小文件浪费空间 |
| 较小 | 空间利用率高，小文件浪费少 | 文件块数多，索引开销大，读写次数多 |

这也是 2019-2020 期末文件系统题中“块大小与空间利用率”计算的核心。

#### 4. LFS：基于日志结构的文件系统（了解）

LFS（Log-structured File System）的基本思路是把磁盘看成一个日志系统，所有写操作都追加到日志头部，尽量把随机写变成顺序写。

| 项目 | 内容 |
| --- | --- |
| 背景 | 缓存使许多读操作不访问磁盘，磁盘负载更多体现为写操作 |
| 核心思想 | 新数据块和元数据先放入缓存，再成段顺序写入磁盘 |
| 优点 | 减少寻道，提高写吞吐 |
| 问题 | segment 会碎片化，需要 cleaning 清理 |
| 恢复 | 使用 checkpoint 和 roll-forward，恢复速度快 |

#### 2. VFS 虚拟文件系统

不同系统可能同时使用多种文件系统，如 Ext4、XFS、FAT、NTFS、APFS。为了屏蔽异构性，操作系统引入 VFS。

VFS：Virtual File System

中间层，对上提供 POSIX API，对下对接不同的文件系统驱动。

```text
用户程序
  │ open/read/write
  ▼
VFS 虚拟文件系统
  │
  ├─ Ext4 驱动
  ├─ XFS 驱动
  ├─ FAT 驱动
  └─ NTFS 驱动
```

VFS 的功能：

| 功能 | 说明 |
| --- | --- |
| 统一接口 | 对所有文件系统提供相同接口 |
| 管理通用对象 | inode、目录项、文件对象、超级块等 |
| 高效查询 | 支持路径遍历和目录项缓存 |
| 调用具体实现 | 根据文件所在文件系统调用对应驱动 |

### 📝 十三、考试指南：文件管理常见题型

#### 题型 1：索引结构支持的最大文件大小【高频】

##### 基本公式

设：

| 符号 | 含义 |
| --- | --- |
| B | 磁盘块大小，单位字节 |
| A | 每个块地址大小，单位字节 |
| N | 一个索引块可存放的地址项数 |

则：

$$
N=\frac{B}{A}
$$

若 inode 中有：

- `d` 个直接索引；
- `s` 个一级间接索引；
- `t` 个二级间接索引；
- `u` 个三级间接索引；

则最大文件大小为：

$$
(d+sN+tN^2+uN^3)\times B
$$

##### 解题步骤

```text
第一步：计算一个索引块能放多少个块地址 N = 块大小 / 地址大小
        │
        ▼
第二步：计算直接索引能表示多少数据块
        │
        ▼
第三步：计算一级、二级、三级间接索引能表示多少数据块
        │
        ▼
第四步：总数据块数 × 块大小
        │
        ▼
第五步：按题目要求换算为 KB / MB / GB
```

##### 例题 1：2024 期末同型题

题目：磁盘块大小为 512B，块地址占 4B，inode 中使用 8 个直接块地址、4 个一次间接块地址和 1 个二次间接块地址，求最大文件大小。

第一步：计算每个索引块可存放地址项数：

$$
N=\frac{512}{4}=128
$$

第二步：分别计算：

| 索引类型 | 数据块数 | 容量 |
| --- | ---: | ---: |
| 直接块 | 8 | $8\times512B=4KB$ |
| 一次间接块 | $4\times128$ | $4\times128\times512B=256KB$ |
| 二次间接块 | $1\times128\times128$ | $128^2\times512B=8192KB$ |

所以：

$$
\text{最大文件大小}=4KB+256KB+8192KB=8452KB
$$

##### 例题 2：2021 期末同型题

题目：块大小 512B，块地址 4B，每个 inode 包括 10 个直接索引、1 个一级间接索引、1 个二级间接索引和 1 个三级间接索引，求最大文件大小。

$$
N=\frac{512}{4}=128
$$

总数据块数：

$$
10+128+128^2+128^3
$$

最大文件大小：

$$
(10+128+128^2+128^3)\times512B
$$

换算为 KB：

$$
(10+128+16384+2097152)\times0.5KB=1056837KB
$$

##### 例题 3：PPT 思考题

题目：系统采用混合索引分配，`iaddr(0)~iaddr(11)` 共 12 个地址项，其中 `iaddr(0)~iaddr(9)` 为直接地址，`iaddr(10)` 为一级间址，`iaddr(11)` 为二级间址。盘块大小 8KB，每个盘块号 4B，求最大文件长度。

第一步：每个索引块可存放：

$$
N=\frac{8KB}{4B}=\frac{8192B}{4B}=2048=2K
$$

第二步：分别计算：

| 类型 | 数据块数 | 容量 |
| --- | ---: | ---: |
| 10 个直接地址 | 10 | $10\times8KB=80KB$ |
| 1 个一级间址 | $2K$ | $2K\times8KB=16MB$ |
| 1 个二级间址 | $2K\times2K$ | $2K\times2K\times8KB=32GB$ |

所以最大文件长度：

$$
80KB+16MB+32GB
$$

若换算为字节：

$$
10\times8192+2048\times8192+2048^2\times8192
$$

#### 题型 2：基于 inode 的路径查找过程【高频】

常见问法：给定路径 `/tmp/foo` 或 `/foo/bar`，说明从根目录开始查找并读取文件的主要步骤。

##### 答题模板

```text
1. 从根目录 inode 开始，读取根目录的数据块；
2. 在根目录内容中查找目标一级目录项，得到对应 inode 号；
3. 读取该目录 inode，得到该目录的数据块地址；
4. 读取该目录数据块，继续查找下一层目录项；
5. 找到目标文件目录项后，得到目标文件 inode 号；
6. 读取目标文件 inode，获得文件大小、权限和数据块指针；
7. 根据 inode 中的数据块指针读取文件实际内容。
```

💡 **注意：** 如果题目说“根目录内容已在内存”，则不用计算读取根目录内容的磁盘访问；如果说“根目录 inode 已读入内存”，仍可能需要读取根目录数据块。

#### 题型 3：读取文件需要访问磁盘几次

2022 期末出现过类似题：只有根目录内容已读入内存，每级目录的目录项都位于一个数据块中，读取 `/tmp/test/helloworld.c` 这样的文件需要访问磁盘几次。

##### 解题思路

```text
第一步：分清题目已在内存的内容
        │
        ├─ 根目录内容已在内存：不用读根目录数据块
        └─ 根目录 inode 已在内存：可能还要读根目录数据块
        │
        ▼
第二步：路径每深入一级目录，通常要读目录 inode 和目录内容
        │
        ▼
第三步：读到目标文件目录项后，还要读目标文件 FCB / inode
        │
        ▼
第四步：根据文件大小和块大小，计算需要读多少个文件数据块
```

如果题目说明“FCB 全部内容直接存储在目录项中”，则找到目录项后不需要额外读取 FCB / inode。

若文件大小为 102KB，数据块大小为 4KB，则文件数据块数：

$$
\left\lceil\frac{102KB}{4KB}\right\rceil=26
$$

然后再加上路径解析过程中读取目录块的次数。

#### 题型 4：块大小与磁盘空间利用率

2019-2020 期末出现过：一半文件大小 2KB，另一半 1.5KB，分别在数据块大小 2KB 和 1KB 时计算空间利用率。

##### 解题模板

```text
第一步：对每类文件分别计算占用块数 = ceil(文件大小 / 块大小)
第二步：占用空间 = 占用块数 × 块大小
第三步：空间利用率 = 文件真实大小 / 实际占用空间
第四步：若有多类文件，按数量比例加权计算
```

##### 示例分析

若块大小为 2KB：

| 文件真实大小 | 实际占用 | 利用率 |
| ---: | ---: | ---: |
| 2KB | 2KB | 100% |
| 1.5KB | 2KB | 75% |

一半文件 2KB，一半文件 1.5KB，则总体利用率：

$$
\frac{2KB+1.5KB}{2KB+2KB}=\frac{3.5}{4}=87.5\%
$$

若块大小为 1KB：

| 文件真实大小 | 实际占用 | 利用率 |
| ---: | ---: | ---: |
| 2KB | 2KB | 100% |
| 1.5KB | 2KB | 75% |

总体仍为：

$$
\frac{2KB+1.5KB}{2KB+2KB}=87.5\%
$$

但一般情况下，块越小，小文件内部碎片越少；块越大，顺序读写效率更高。

#### 题型 5：大文件随机读取性能下降原因

常见答题点：

1. 大文件可能需要间接索引，访问数据块前要先访问一级/二级/三级索引块；
2. 随机读取局部性差，缓存命中率低；
3. 文件数据块可能分散在磁盘各处，机械磁盘上寻道和旋转延迟增加；
4. 大文件索引结构更复杂，路径为“索引块 → 数据块”，I/O 次数更多。

解决思路：

| 方法 | 作用 |
| --- | --- |
| 增大缓存 | 提高索引块和热点数据块命中率 |
| 预读 / 聚簇读取 | 对有局部性的访问提前读入相邻块 |
| 优化文件布局 | 尽量连续或局部集中分配数据块 |
| 使用 extent | 用“起始块 + 长度”表示连续区间，减少索引项 |
| 使用 SSD | 降低随机访问的寻道/旋转代价 |

#### 题型 6：文件物理结构选择题

常见判断：

| 判断点 | 正确结论 |
| --- | --- |
| 连续文件是否支持随机访问 | 支持，逻辑块到物理块可直接计算 |
| 连续文件是否利于动态增长 | 不利于 |
| 链接文件随机访问效率 | 很低 |
| 链接文件是否适合动态增长 | 适合 |
| 索引文件是否支持随机访问 | 支持 |
| 索引文件代价 | 索引表占空间，访问索引块耗时 |
| 目录是否是文件 | 是，目录是一种特殊文件 |

### ✅ 本章速记总结

1. 文件是带符号名的字节序列，是操作系统为持久化存储、共享和抽象外存而提供的机制。
2. 文件由文件体和文件说明组成；文件说明即元数据，如大小、位置、权限、时间戳等。
3. 文件系统负责命名、目录、空间分配、文件读写、保护共享、可靠性和性能优化。
4. 目录是特殊文件，内容是目录项；多级目录能解决重名、分组和管理问题。
5. FCB / inode 保存文件属性和数据块指针；目录项通常保存“文件名 + inode 号”。
6. 连续结构访问快但不利于增长；链接结构易增长但随机访问差；索引结构既支持顺序访问又支持随机访问，但有索引开销。
7. 综合索引模式通过直接索引服务小文件，通过一级、二级、三级间接索引支持大文件。
8. 空闲空间管理方法包括空闲表、位图、空闲链表和成组链接法。
9. 文件保护包括副本、定时转储、权限控制和一致性检查；并发访问可通过文件锁协调。
10. 文件系统性能优化重点是减少磁盘 I/O：块高速缓存、预读、延迟写、优化布局和选择合适块大小。
11. 期末高频题包括：索引结构最大文件大小、inode 路径查找过程、文件读取磁盘访问次数、块大小与空间利用率、文件物理结构优缺点比较。
