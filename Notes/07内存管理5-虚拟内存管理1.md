# 📚 虚拟内存管理

---

## 模块零：前置思想 —— 如何突破物理内存容量限制？

要解决"1G内存如何跑2G程序"的问题，核心思想是**内存的"时空复用"**。早期有两种实现思路：

### 0.1 覆盖技术 (Overlay)
*   **核心思想**：将**程序**按**调用关系分段**，不会**同时用到的模块**共享同一块内存区域。
*   **空间分配原则**：一个覆盖段分配的内存大小，等于该段中**最大模块的大小**。
*   **致命缺点**：**对程序员不友好（不透明）**。程序员必须**手动分析调用关系**并划分覆盖段，极其繁琐，现已基本淘汰。

### 0.2 交换技术 (Swapping) 💡[虚拟内存的核心机制基础]
*   **核心思想**：把暂时不用的数据或进程换出到辅存（磁盘），腾出空间给需要的进程，用时再换回。
*   **与覆盖的本质区别**：**交换对程序员是完全透明的**，由操作系统自动在后台完成。
*   **交换策略**：换出"最近不会用到"的进程（如后台等待I/O的进程）；**内核核心代码决不能换出**。
    *   **积极策略 vs 懒惰策略**：懒惰策略就是**直到内存不够时，再交换**。

## 模块一：局部性原理 💡[虚拟存储器的理论基石，必考重点]

### 1.1 定义
程序在执行过程中的较短时期内，所执行的指令地址和操作数地址分别局限于一定区域。

### 1.2 两种表现形式
| 局部性类型 | 说明 | 举例 |
|-----------|------|------|
| **时间局部性** | 最近被访问过的数据/指令，很快很可能再次被访问 | 循环结构、频繁读写的变量 |
| **空间局部性** | 某个存储单元被访问，其**附近的单元**也很可能马上被访问 | 数组的顺序遍历、代码顺序执行 |

### 1.3 典型性能优化案例
#### 排序算法性能差异
实际运行时间：堆排序 >> 快排
* 原因：堆结构的下沉与上浮往往在非连续空间内读取，局部性差，Cache命中率低。

#### 矩阵乘法循环顺序优化
C语言中数组是按行存储的。在矩阵乘法 $C = A \times B$ 中：

**IJK版本（一列乘一行，产生一个点）：**
```c
for (i = 0; i < N; i++)
    for (j = 0; j < N; j++) {
        C[i][j] = 0;
        for (k = 0; k < N; k++)
            C[i][j] += A[i][k] * B[k][j]; // 列跳跃访问，Cache命中率极低
    }
```

**KIJ版本（一行乘一列，Cache友好）：**
```c
for (k = 0; k < N; k++)
    for (i = 0; i < N; i++) {
        r = A[i][k]; // 把A的数据放在局部变量寄存器，防止多次读入
        for (j = 0; j < N; j++)
            C[i][j] += r * B[k][j]; // 同行存储位置邻近，Cache命中率高
    }
```
> 优化技巧：对于连续多次使用到的同一个值，把它存在临时变量（寄存器）中，加快读取速度。两种写法性能差异可达数倍！

## 模块二：虚拟存储器概述

### 2.1 常规存储管理的问题
| 特征 | 说明 |
|------|------|
| **一次性** | 要求作业全部装入内存后方能运行 |
| **驻留性** | 作业装入内存后一直驻留内存，直至结束 |

**后果**：大作业无法运行，并发度受限。

### 2.2 虚拟存储器的定义 ★考试必考
在具有层次结构存储器的计算机系统中，采用自动实现**部分装入**和**部分对换**功能，为用户提供一个比物理内存容量大得多的可寻址存储器。

> **核心能力**：给每个进程提供一个**大的、一致的、连续可用的和私有的地址空间**
> - **一致性**：所有进程看到统一的地址空间布局
> - **保护性**：隔离进程地址空间，防止互相破坏
> - **高效性**：主存作为磁盘的高速缓存

## 模块三：虚拟存储的核心概念 💡[简答/辨析重点]

在之前学过的"实存管理"（如基本分页、分段）中，程序必须**一次性全部装入内存**才能运行。虚拟存储打破了这一限制：**程序只把现在要用的部分装入，运行中需要哪部分再动态调入**。

## 模块四：虚拟存储的核心原理

### 3.1 基本工作流程
1. **装入时**：只需将当前需要执行的部分页/段读入内存即可开始执行
2. **执行中**：缺页/缺段时，由OS将相应页/段调入内存，继续执行
3. **置换**：将内存中暂时不使用的页/段调出到外存，腾出空间

### 3.2 虚存管理 vs 对换技术 ★辨析题
| 维度 | 对换技术 | 虚存管理 |
|------|---------|---------|
| **单位** | 以进程为单位 | 以页或段为单位 |
| **限制** | 进程内存 > 系统内存时无法运行 | 即使进程内存 > 系统内存仍能运行 |
| **本质** | 进程级的交换 | 内存资源的时分复用 |

### 3.3 虚拟存储的四大特征 ★简答题必背
| 特征 | 说明 | 地位 |
|------|------|------|
| **离散性** | 物理分配不连续，虚拟地址空间使用也不连续 | 基础 |
| **多次性** | 作业被分成多次调入内存运行 | **最重要特征**，实现逻辑扩容 |
| **对换性** | 运行中可换进换出，提高内存利用率 | 基础 |
| **虚拟性** | 从逻辑角度访问，范围大但不超过物理内存+外存容量 | 最终效果 |

> **依存关系**：虚拟性以多次性和对换性为基础，后两者以离散分配为基础

## 模块五：虚拟内存 vs Cache ★对比分析题

### 5.1 相同点
- **出发点相同**：都是为了提高存储系统的性能价格比
- **原理相同**：都利用局部性原理，将最近常用信息从慢速大容量调入高速小容量

### 5.2 不同点 ★★★
| 维度 | Cache | 虚拟内存 |
|------|-------|---------|
| **侧重点** | 解决主存与CPU的**速度差异** | 解决**存储容量问题**，还包括存储管理、分配、保护 |
| **数据通路** | CPU与cache和主存均有直接通路，cache不命中可直接访存 | 辅存与CPU无直接通路，主存不命中只能调页 |
| **透明性** | **完全硬件实现**，对所有程序员均透明 | OS与硬件共同完成，对系统程序员**不透明** |
| **未命中损失** | 主存是cache的5~10倍 | 主存比辅存快**上千倍**，损失远大于cache未命中 |

## 模块六：请求分页存储管理

### 6.1 请求分页系统定义
在分页系统基础上，增加**请求调页功能**和**页面置换功能**形成的页式虚拟存储器系统。

### 6.2 硬件支撑
| 类别 | 功能 |
|------|------|
| **MMU（内存管理单元）** | 完成逻辑地址到物理地址转换，管理页表基址，分解地址，管理TLB，发出缺页中断 |
| **扩展页表** | 传统页表增加标志位 |

### 6.3 页表项扩展标志位 ★选择/填空必考
| 标志位 | 作用 |
|--------|------|
| **驻留位（Present）** | 该页存在物理内存还是磁盘（虚存最重要的位） |
| **修改位（Modified/Dirty）** | 页面在内存中是否被修改过，决定换出时是否需要写回磁盘 |
| **引用位（Referenced/Accessed）** | 追踪页是否被访问（读/写），为置换算法提供依据 |
| **保护位（Protected）** | 控制读/写/执行权限 |

### 6.4 调入策略
| 策略 | 定义 | 特点 |
|------|------|------|
| **预调页（Prepaging）** | 同时将预计需要的所有页一起调入 | 阻止大量页错误，但可能调入无用页面，成本不一定低 |
| **按需调页（Demand Paging）** | 当且仅当需要某页时才调入 | 被虚拟内存系统普遍采用，使用懒惰交换（lazy swapper） |

## 模块七：缺页错误处理机制 ★简答/综合题

### 7.1 完整处理流程（9步）★必考
1. **现场保护**：陷入内核态，保存OS及用户进程状态信息
2. **页面定位**：查找发生中断的虚拟页面（硬件寄存器或软件分析指令）
3. **权限检查**：检查虚拟地址有效性及保护位，非法则杀死进程
4. **找空闲页框**：查找空闲物理页框，无空闲则调用置换算法选淘汰页
5. **旧页面写回**：若淘汰页被修改过，将内容写回磁盘（页框置忙）
6. **新页面调入**：将磁盘上的新页面内容复制到页框（发生上下文切换）
7. **更新页表**：磁盘中断到来后，更新页表项，标记页框为正常状态
8. **恢复现场**：恢复缺页中断前的状态，程序指针指向引发缺页的指令
9. **继续执行**：重新执行引发缺页中断的指令

> **代价极高**：磁盘访问ms级 vs 内存访问ns级，相差6个数量级！

## 模块八：页面置换算法 ★计算/分析题必考

### 8.1 各算法对比总结
| 算法 | 核心思想 | 优点 | 缺点 | 特殊性质 |
|------|---------|------|------|---------|
| **OPT（最优）** | 置换未来最久不用的 | 缺页率最低 | **无法实现**（需预知未来） | 作为衡量基准 |
| **FIFO（先进先出）** | 置换最早进入的 | 实现最简单 | 性能差，常用页可能被反复换出 | **存在Belady现象** |
| **Clock（时钟）** | 环形队列+访问位 | 改进FIFO，无移动开销 | 命中率仍不高 | Second Chance的改进版 |
| **LRU（最近最少使用）** | 置换最久未使用的 | 接近最优 | 硬件开销大 | 无Belady现象，栈算法 |
| **Aging（老化）** | 移位寄存器近似LRU | 硬件可行，接近LRU | 有精度损失 | 实用版LRU |
aging： n位老化寄存器

### 8.2 Belady现象 ★名词解释/辨析
**定义**：使用FIFO算法时，**分配的页面增多，但缺页率反而提高**的异常现象。

**典型示例**：访问序列 1,2,3,4,1,2,5,1,2,3,4,5
- 3个页面时：缺页9次
- 4个页面时：缺页10次（反而增加！）

**本质原因**：FIFO的置换特征与进程访问内存的动态特征矛盾。

> **注意**：栈算法（如LRU）不会出现Belady现象。

### 8.3 Clock算法（最近未使用NRU）
- **改进点**：用环形队列避免Second Chance中页面移动开销
- **规则**：缺页时检查指针指向的页面，访问位为1则清0并移动指针；为0则替换该页

## 模块九：抖动（Thrashing）与工作集模型 ★选择/简答

### 9.1 抖动（Thrashing）问题

#### 1. 什么是抖动？
**定义**：页面频繁地换入换出，导致CPU大部分时间都在处理缺页中断，真正用于执行用户程序的时间极少，系统性能急剧下降的现象。

#### 2. 抖动的根本原因
**进程分配到的物理页框数，小于进程正常运行所需的"工作集"大小**。

局部性原理失效，进程访问的每一页几乎都不在内存，执行一条指令可能要触发好几次缺页，大部分时间都在等磁盘。

#### 3. 导致抖动的常见诱因
| 诱因 | 说明 |
|------|------|
| **多道程序度过高** | 系统里进程太多，每个进程分到的页框都不够用 |
| **置换算法选择不当** | 比如FIFO把常用页换出去了，刚换出去又要换回来 |
| **程序本身局部性差** | 比如超大数组随机访问，破坏了时间和空间局部性 |

#### 4. 解决抖动的方法
| 方法 | 核心思想 |
|------|---------|
| **挂起部分进程** | 把优先级低的进程换出到外存，腾出页框给剩下的进程 |
| **调整多道程序度** | 限制同时在内存的进程数量 |
| **改进置换算法** | 用LRU类算法减少误置换 |
| **工作集模型** | ✅ 最根本的解决方法：根据工作集大小分配页框 |

---

### 9.2 工作集模型（Working Set Model）

#### 1. 核心思想
基于**局部性原理**：进程在一段时间内只会访问一小部分页面。我们把这部分页面称为"工作集"，只要能把工作集全部装进内存，进程就能顺畅运行，几乎不会缺页。

#### 2. 工作集的定义
```
W(t, Δ) = 进程在时间 (t - Δ, t) 内访问过的页面集合
```
- `t` = 当前时间
- `Δ` = 工作集窗口大小（Working Set Window）
- 窗口大小Δ是关键参数，决定了工作集的大小

#### 3. 工作集的关键特性
1.  **动态变化**：进程的工作集是随时间变化的，不是固定的
2.  **时间窗口敏感**：窗口太小，工作集装不下；窗口太大，工作集包含太多不用的页
3.  **阶段突变**：程序进入新的执行阶段时（比如从初始化进入主循环），工作集会突然改变，出现"工作集突变"

#### 4. 工作集模型的应用原则
```
如果所有进程的工作集大小之和 ≤ 系统总页框数 → 系统可以顺畅运行
如果超过了 → 必然发生抖动，必须挂起部分进程
```

> 💡 这是虚拟内存调度的黄金准则：不要让系统中所有进程的工作集总和超过物理内存容量。

#### 5. 工作集置换算法
**核心规则**：置换那些**不在当前工作集中**的页。

实现思路：
- 记录每个页最后一次被访问的时间
- 如果 `当前时间 - 最后访问时间 > Δ` → 说明这个页不在工作集中，可以置换

#### 6. 常用变种：工作集时钟算法（WSClock）
把Clock算法和工作集结合，是真实操作系统常用的算法：
- 环形队列扫描
- 除了访问位，还要检查页面的"年龄"
- 年龄超过Δ窗口且访问位为0的页，优先置换

> 📌 典型简答题：
> **Q：为什么增加多道程序度反而可能降低CPU利用率？**
> A：当多道程序度过高时，每个进程分配到的页框不足以容纳其工作集，发生抖动现象。CPU大部分时间用于处理缺页中断，真正执行用户程序的时间减少，导致CPU利用率不升反降。
