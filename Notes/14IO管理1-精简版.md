### 📚 一、I/O 管理概述：为什么 OS 要管理 I/O？

#### 1. 核心问题

I/O 设备种类多、速度差异大、控制接口复杂，常成为系统性能瓶颈。OS 需要在用户进程与硬件之间提供统一抽象，完成设备访问、控制、分配、缓冲、调度与错误处理。

**bus：系统总线**。总线是接入 I/O 设备的重要方式：

$$
总线带宽 = 频率 \times 宽度\ \text{byte/sec}
$$

键盘约 `10B/s`，PCI 总线可达数百 `MB/s`。这种速度差异决定了 I/O 管理必须使用缓冲、异步与并行机制。

#### 2. 设备管理目标与功能

| 目标 | 含义 |
| --- | --- |
| 提高效率 | 匹配 CPU 与外设速度差异，提高 I/O 吞吐 |
| 方便使用 | 提供统一接口，屏蔽硬件细节 |
| 方便控制 | 支持设备增加、删除、控制和错误处理 |
| 提高并行性与安全性 | 让 CPU 与设备并行工作，并保护传输数据 |

| 功能 | 说明 |
| --- | --- |
| 用户接口 | 命令接口、编程接口、系统调用 |
| 设备分配和释放 | 分配设备、控制器、通道等资源 |
| 设备访问和控制 | 并发访问、设备控制、差错处理 |
| I/O 缓冲和调度 | 通过缓冲、调度提高效率 |
| 中断处理 | I/O 完成后通知 CPU |

#### 3. I/O 设备分类

| 分类依据 | 类型 | 典型设备 | 特点 |
| --- | --- | --- | --- |
| 传输速度 | 低速 / 高速 / 更高速 | 键盘、鼠标 / 磁盘、网卡 / 总线 | 速度差异决定缓冲和控制方式 |
| 信息交换单位 | 块设备 | 磁盘、磁带、光驱 | 可寻址，支持随机读写，速率较高 |
| 信息交换单位 | 字符设备 | 键盘、鼠标、串口 | 通常不可寻址，速率较低 |
| 信息交换单位 | 网络设备 | 以太网、WiFi、蓝牙 | 以包/帧通信，依赖网络地址 |
| 共享属性 | 独占设备 | 打印机、磁带机 | 一段时间只允许一个进程使用 |
| 共享属性 | 共享设备 | 硬盘 | 多进程可交叉访问 |
| 共享属性 | 虚设备 | SPOOLing 打印机 | 用软件模拟共享设备 |

### 🏗️ 二、I/O 硬件机制：CPU 如何控制设备？

#### 1. I/O 设备组成

I/O 设备通常由两部分组成：

1. **机械部分**：设备本身；
2. **电子部分**：设备控制器或适配器。

设备控制器连接 CPU 与具体设备，是 OS 控制外设的关键抽象。

#### 2. 设备控制器与寄存器

设备控制器通过寄存器与 CPU 交互：

| 寄存器 | 作用 |
| --- | --- |
| 数据寄存器 | 暂存 CPU 与设备交换的数据 |
| 控制寄存器 | 接收 CPU 发来的控制命令 |
| 状态寄存器 | 报告忙、闲、完成、出错等状态 |

设备控制器功能：接收并识别 CPU 命令；完成 CPU、控制器、设备之间的数据交换；报告设备状态；识别设备地址；提供硬件缓冲；进行差错检测。

#### 3. I/O 端口地址与寄存器寻址

I/O 端口地址是接口电路中每个寄存器的唯一地址。CPU 访问设备寄存器主要有两种方式：

| 编址方式 | 说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 内存映像编址（MMIO） | 将设备寄存器映射到物理地址空间，用 load/store 访问 | 指令统一，保护机制自然 | 设备寄存器不能被 cache |
| I/O 独立编址（PMIO） | I/O 有独立地址空间，如 Intel `in/out` 指令 | 不占内存地址空间，易区分 I/O | 指令少，不灵活 |

许多 RISC 架构常用内存映射 I/O：设备寄存器与内存复用地址总线，通过特定物理地址范围访问。

#### 4. 为什么设备寄存器不能走 cache？

设备寄存器反映硬件实时状态，不能被 cache。例如 MIPS 中不走 cache 的地址段对应 `kseg1`。如果状态寄存器第一次读取后进入 cache，后续 CPU 可能一直读到旧值，无法正确判断设备是否完成或出错。

```text
CPU 读取设备状态寄存器
        │
        ▼
如果走 cache：
        ├─ 第一次读取：从设备寄存器取值，并缓存
        └─ 后续读取：可能直接读 cache 旧值 ❌

正确做法：
        ▼
设备寄存器所在地址区设置为不可缓存
每次都直接访问真实设备状态 ✅
```

### 🚀 三、I/O 控制方式：CPU 干预如何逐步减少？

【四种控制方式（流程，是什么）， CPU干预程度对比，有什么样的特点，适合于什么样的场景，不适用于什么场景】

四种方式按 CPU 干预程度由高到低为：**PIO（轮询）→ 中断驱动 → DMA → 通道**。

#### 1. PIO：程序控制 I/O（轮询方式）

PIO（Programmed I/O）中，CPU 发出命令后不断测试设备状态，并由 CPU 搬运数据。适合简单、低速、小数据量设备。

![1](assets/image-30.png)

```text
应用程序提出 I/O 请求
        │
        ▼
设备驱动检查设备状态
        │
        ▼
设备正常 → 发出控制命令
        │
        ▼
CPU 不断轮询设备状态
        ├─ 未完成：继续等待/查询
        └─ 已完成：取回数据
        ▼
应用程序继续执行
```

| 方面 | 内容 |
| --- | --- |
| CPU 干预程度 | 最高，CPU 全程轮询和搬运数据 |
| 优点 | 硬件简单，编程容易 |
| 缺点 | CPU 忙等，浪费时间；开销与数据量成正比 |
| 适合场景 | 简单、低速、小数据量 I/O |
| 不适合场景 | 高速设备、大批量传输 |

#### 2. 中断驱动 I/O

中断驱动方式中，CPU 发出 I/O 命令后记录请求状态并继续执行其他工作；设备完成后主动中断 CPU。中断处理程序识别完成信号，设备管理程序查询设备状态表，提交结果、传回数据或唤醒进程。

![31](assets/image-31.png)

需要维护**设备状态表**，记录 I/O 请求与执行状态。

```text
用户程序提出 I/O 请求
        │
        ▼
设备驱动检查设备状态并发出命令
        │
        ▼
将请求状态记录到设备状态表
        │
        ▼
CPU 继续执行其他任务
        │
        ▼
设备完成 I/O 后发出中断
        │
        ▼
中断处理程序识别完成信号
        │
        ▼
设备管理程序查询设备状态表
        │
        ▼
提交结果 / 传回数据 / 唤醒进程
```

| 方面 | 内容 |
| --- | --- |
| CPU 干预程度 | 中等，发起和完成时需要 CPU，中间可并行 |
| 优点 | 避免忙等，提高 CPU 利用率 |
| 缺点 | 中断需保护/恢复现场；频繁中断开销大 |
| 适合场景 | 低中速、事件驱动明显的设备 |
| 不适合场景 | 每字节/字符都中断的高速大数据传输 |

#### 3. DMA：直接存储访问方式

DMA（Direct Memory Access）让 DMA 控制器负责内存与外设之间的数据块传输。CPU 只在开始前设置参数，完成后响应中断。

| DMA 寄存器 | 作用 |
| --- | --- |
| 命令/状态寄存器 CR | 接收 CPU 命令，保存控制信息或设备状态 |
| 内存地址寄存器 MAR | 保存内存起始地址 |
| 数据寄存器 DR | 暂存设备与内存之间传输的数据 |
| 数据计数器 DC | 保存传输字节数或字数 |

```text
CPU 设置 DMA 控制器参数
  ├─ 内存起始地址 MAR
  ├─ 传输字节数 DC
  ├─ 传输方向
  └─ 控制命令 CR
        │
        ▼
CPU 发起 I/O 操作后继续执行其他任务
        │
        ▼
DMA 控制器接管总线
        │
        ▼
设备 ↔ 内存 成批传输数据
        │
        ▼
传输完成后 DMA 控制器发出中断
        │
        ▼
CPU 响应中断并后续处理
```

读取磁盘数据例子：

```text
1. 设备驱动收到读取磁盘数据到内存地址 X
2. 设备驱动控制磁盘控制器从磁盘读取数据
3. 磁盘控制器初始化 DMA 传送
4. 磁盘控制器传送数据到 DMA 控制器
5. DMA 控制器传送 C 字节数据到内存地址 X
6. DMA 控制器完成后产生中断，通知 CPU
```

| 方面 | 内容 |
| --- | --- |
| CPU 干预程度 | 较低，只在开始和结束时干预 |
| 优点 | 成批数据不需 CPU 搬运，适合高速设备 |
| 缺点 | 方向、地址、长度仍需 CPU 设置；可能需要更多 DMA 控制器 |
| 适合场景 | 磁盘、网卡等高速块传输设备 |
| 不适合场景 | 极简单设备或小数据量 I/O |

#### 4. 通道控制方式（Channel）

通道是专门负责 I/O 的处理器，有自己的指令和程序，可执行由通道指令组成的通道程序，进一步减少 CPU 干预。

```text
CPU 构造通道程序
        │
        ▼
CPU 启动通道
        │
        ▼
通道执行通道程序
        ├─ 控制多个 I/O 操作
        ├─ 管理设备传输
        └─ 与 CPU 分时使用内存
        │
        ▼
完成后中断通知 CPU
```

| 通道类型 | 工作方式 | 适用设备 |
| --- | --- | --- |
| 字节多路通道 | 以字节为单位交叉服务多个设备 | 打印机、终端等低/中速设备 |
| 数组选择通道 | 每次为一台设备传送一批数据 | 磁盘、磁带等高速设备 |
| 数组多路通道 | 分时交叉 + 高速批量传输 | 多个高速设备并行场景 |

#### 5. DMA 与中断方式区别

| 比较点 | 中断驱动方式 | DMA 方式 |
| --- | --- | --- |
| 中断时机 | 每个数据传送完成后可能中断 | 一批数据传输完成后才中断 |
| 数据传送者 | CPU 在中断处理时控制完成 | DMA 控制器直接在主存和设备间传输 |
| CPU 开销 | 频繁中断时较大 | 只负责开始和结束，开销小 |
| 适用对象 | 低中速设备、异常事件处理 | 高速设备、数据块传输 |

#### 6. 四种控制方式综合对比

| 控制方式 | CPU 干预程度 | 数据传输主体 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| PIO/轮询 | 最高 | CPU | 硬件简单 | CPU 忙等 | 简单、低速、小数据量设备 |
| 中断驱动 | 较高 | CPU | CPU 不必忙等 | 中断频繁时开销大 | 低中速、事件驱动设备 |
| DMA | 较低 | DMA 控制器 | 成批传输，CPU 开销小 | 需 DMA 控制器，参数仍需 CPU 设置 | 磁盘、网卡等高速块设备 |
| 通道 | 最低 | 通道处理器 | 可执行通道程序，控制多组 I/O | 硬件复杂、费用高 | 大型机、多设备复杂 I/O |

💡 **线索：** 四种方式体现“CPU 逐步摆脱 I/O”：

```text
PIO：CPU 一直等
        ▼
中断：CPU 不等，但数据仍由 CPU 处理
        ▼
DMA：一批数据由 DMA 搬运
        ▼
通道：连传输控制逻辑也交给专用处理器
```

### 🔄 四、I/O 软件层次：从用户请求到硬件执行

#### 1. 驱动程序本质

驱动程序把设备文件的读写转化为对数据寄存器、控制寄存器、状态寄存器的读写。用户看到文件或统一接口；驱动面向具体设备控制器，真正控制硬件。

#### 2. I/O 软件层次 【重点！】

![29](assets/image-29.png)

| 层次 | 名称 | 功能 |
| --- | --- | --- |
| 4 | 用户进程 | 使用 I/O 系统调用，进行格式化 I/O |
| 3 | 设备无关软件 | 命名、保护、阻塞、缓冲、设备分配 |
| 2 | 设备驱动程序 | 设置设备寄存器，检测设备状态 |
| 1 | 中断处理程序 | I/O 结束时处理中断，唤醒设备服务子程序 |
| 0 | 硬件 | 执行实际 I/O 操作 |

#### 3. I/O 请求处理过程

```text
用户进程
  │ 发出 read/write 等 I/O 请求
  ▼
系统调用进入内核
  │
  ▼
设备无关软件
  ├─ 逻辑设备名转换
  ├─ 权限保护
  ├─ 缓冲管理
  └─ 设备分配
  │
  ▼
设备驱动程序
  ├─ 检查设备状态寄存器
  ├─ 写控制寄存器发出命令
  └─ 准备数据寄存器或 DMA 参数
  │
  ▼
硬件设备执行 I/O
  │
  ▼
完成后产生中断
  │
  ▼
中断处理程序保存结果并通知驱动
  │
  ▼
驱动/内核 I/O 子系统唤醒进程并返回结果
```

#### 4. 设备独立性

设备独立性是指用户程序使用逻辑设备名，系统执行时将逻辑设备名映射到物理设备名。

| 概念 | 含义 |
| --- | --- |
| 逻辑设备 | 用户程序看到的设备名，如 `/dev/printer` |
| 物理设备 | 实际硬件设备，如某台具体打印机 |
| LUT | 逻辑设备表，用于逻辑设备名到物理设备名的映射 |

| 逻辑设备名 | 物理设备名 | 驱动程序入口地址 |
| --- | --- | --- |
| `/dev/tty` | 3 | 1024 |
| `/dev/printer` | 5 | 2046 |

#### 5. 设备驱动程序组成与特点

| 组成 | 作用 |
| --- | --- |
| 自动配置和初始化子程序 | 检测设备是否存在、是否正常，并初始化状态 |
| I/O 操作子程序 | 响应系统调用，执行具体 I/O |
| 中断服务子程序 | 接收硬件中断，完成善后处理，唤醒阻塞进程 |

特点：内核的一部分，错误可能导致系统严重问题；与设备特性和 I/O 控制方式紧密相关；向内核提供标准接口；可动态加载和卸载。

### ⚖️ 五、I/O 缓冲管理：匹配速度差异、减少中断、提高并行性

#### 1. 为什么需要缓冲？

缓冲技术通过在设备与进程之间设置缓冲区组织数据传输，直接目标是**提高外设利用率**。主要目的：

- 匹配 CPU 和外设的不同处理速度；
- 减少 CPU 中断次数；
- 提高 CPU 与 I/O 外设的**并行性**。

缓冲把零散数据分组，使 CPU 处理、设备输入、数据复制尽可能重叠，提高**吞吐量**。

#### 2. 缓冲代价：多次复制

缓冲可能造成数据包在设备、内核缓冲区与用户区之间多次复制。

<img src="./assets/image-20260611001523908.png" alt="image-20260611001523908" style="zoom: 67%;" />

```text
发送方向：用户空间 → 内核空间 → 网络控制器 → 网络
接收方向：网络 → 网络控制器 → 内核空间 → 用户空间
```

#### 3. 单缓冲

单缓冲只有一个缓冲区，CPU 与外设轮流使用。设：

| 符号 | 含义 |
| --- | --- |
| T | 设备输入数据到缓冲区的时间 |
| M | 从缓冲区传送数据到用户工作区的时间 |
| C | 用户进程处理数据的时间 |

由于 `T` 和 `C` 可以并行，单缓冲每块数据处理时间为：

$$
\max(C,T)+M
$$

![image-20260611125704688](./assets/image-20260611125704688.png)

```text
第 i 块数据：设备输入 T → 缓冲区复制到用户区 M → CPU 处理 C
流水效果：CPU 处理第 i 块时，设备可输入第 i+1 块；但只有一个缓冲区，仍受轮流使用限制
```

#### 4. 双缓冲

双缓冲设置两个缓冲区，使设备向一个缓冲区输入时，CPU 可处理另一个缓冲区中的数据。

![image-20260611130322434](./assets/image-20260611130322434.png)

稳态处理一块数据的时间近似为：

$$
\max(C+M,T)
$$

| 情况 | 效果 |
| --- | --- |
| `C < T` | 块设备可连续输入，减少等待 CPU |
| `C > T` | CPU 不必等待设备输入 |

#### 5. 循环缓冲

循环缓冲由多个缓冲区组成环形队列，适用于 CPU 与外设速度差异较大、双缓冲不够的场景。

| 缓冲区类型 | 含义 |
| --- | --- |
| R：空缓冲区 | 可供输入进程写入数据 |
| G：已装满数据的缓冲区 | 可供计算进程读取 |
| C：工作缓冲区 | 计算进程当前正在使用 |

| 指针 | 作用 |
| --- | --- |
| Nexti | 输入进程下次可用的空缓冲区 R |
| Nextg | 计算进程下一个可用的满缓冲区 G |
| Current | 计算进程当前工作缓冲区 C |

```text
          ┌──────────┐
          │ Buffer 1 │
          └────┬─────┘
               │
               ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Buffer 4 │◄─┤ Buffer 2 │─►│ Buffer 3 │
└──────────┘  └──────────┘  └──────────┘
     ▲                         │
     └─────────────────────────┘

输入进程：沿环寻找空缓冲区 R 写入
计算进程：沿环寻找满缓冲区 G 读取
```

#### 6. 缓冲池

大量 I/O 进程若各自使用专用循环缓冲，会浪费内存。缓冲池让多个进程共享一组缓冲区。

| 队列 | 含义 |
| --- | --- |
| 空缓冲队列 emq | 存放空闲缓冲区 |
| 输入队列 inq | 存放装满输入数据的缓冲区 |
| 输出队列 outq | 存放装满输出数据的缓冲区 |

#### 7. 题型：缓冲区时间计算

期末常考磁盘读入缓冲区、缓冲区传送到用户区、CPU 处理三段时间组合。

| 符号 | 含义 |
| --- | --- |
| T | 从磁盘将 1 块数据传送到缓冲区的时间 |
| M | 将缓冲区数据传送到用户区的时间 |
| C | CPU 处理一个数据块的时间 |

单缓冲平均每块处理时间：

$$
\max(C,T)+M
$$

例如 2021 期末：`T=50ms, M=20ms, C=60ms`：

$$
\frac{1000}{\max(60,50)+20}=\frac{1000}{80}=12.5\ \text{块/s}
$$

双缓冲稳态节拍：

$$
\max(T, M+C)
$$

若计算有限块数：先读入第一块需要 `T`，之后按稳态节拍推进，最后还要考虑最后一块传送与处理。

例如 2024 期末：`T=50ms, M=10ms, C=40ms`，双缓冲处理 10 块数据。因为：

$$
M+C=10+40=50\text{ms}=T
$$

流水线每 50ms 完成一块，总时间为：

$$
50+10\times(10+40)=550\text{ms}
$$

### 💡 六、设备管理与 SPOOLing

#### 1. 设备管理关键数据结构

一个完整 I/O 数据通路可能包括：**设备 → 控制器 → 通道**。

##### （1）DCT：设备控制表

每个设备一张 DCT（Device Control Table）。

| 字段 | 说明 |
| --- | --- |
| 设备类型 type | 设备类型标识 |
| 设备标识符 deviceid | 设备唯一标识 |
| 设备状态 | 等待/不等待/忙/闲 |
| 指向控制器表的指针 | 指向连接的控制器 |
| 重复执行次数或时间 | 出错时允许重传次数 |
| 设备队列队首指针 | 等待该设备的进程队列 |

##### （2）COCT：控制器控制表

每个设备控制器一张 COCT，记录控制器标识符、控制器状态、连接的通道表指针、控制器等待队列队首/队尾指针。

##### （3）CHCT：通道控制表

每个通道一张 CHCT，记录通道标识符、通道状态、连接的控制器表首址、通道等待队列队首/队尾指针。

##### （4）SDT：系统设备表

SDT 反映系统中所有设备资源状态，记录所有设备及其 DCT 入口。

| 字段 | 说明 |
| --- | --- |
| DCT 指针 | 指向相应设备的 DCT |
| 设备使用进程标识 | 当前使用设备的进程 |
| DCT 信息 | 保存设备标识、设备类型等 |

#### 2. 设备分配过程

```text
用户请求某设备
        │
        ▼
查找系统设备表 SDT
        │
        ▼
找到设备控制表 DCT
        ├─ 设备忙：进程进入设备等待队列
        └─ 设备闲：分配设备
                 │
                 ▼
           查找控制器控制表 COCT
                 ├─ 控制器忙：进入控制器等待队列
                 └─ 控制器闲：分配控制器
                          │
                          ▼
                    查找通道控制表 CHCT
                          ├─ 通道忙：进入通道等待队列
                          └─ 通道闲：建立完整 I/O 通路 ✅
```

#### 3. 设备分配考虑因素

| 因素 | 内容 |
| --- | --- |
| 设备固有属性 | 独占设备、共享设备、虚拟设备 |
| 分配算法 | 先来先服务、优先级高者优先 |
| 安全性 | 是否可能导致死锁 |

| 类型 | 特点 | 优缺点 |
| --- | --- | --- |
| 安全分配（同步） | 请求 I/O 后阻塞，直到完成 | 可防止死锁，但 CPU 和 I/O 串行，效率低 |
| 不安全分配（异步） | 请求 I/O 后仍可运行并请求其他资源 | 效率高，但需安全性检查，可能死锁 |

#### 4. SPOOLing：把独享设备变成虚拟共享设备

SPOOLing（Simultaneous Peripheral Operations On-Line，假脱机技术）在硬件设备之上增加软件抽象层，把**独享设备**转变为具有**共享特征**的虚拟设备。本质是用共享设备模拟独享设备、用高速设备模拟低速设备。

| 组成 | 作用 |
| --- | --- |
| 输入井 | 磁盘区域，收容输入设备送来的数据 |
| 输出井 | 磁盘区域，收容用户程序输出数据 |
| 输入缓冲区 | 内存缓冲区，暂存输入设备数据 |
| 输出缓冲区 | 内存缓冲区，暂存输出井送往输出设备的数据 |
| 输入进程 SPi | 模拟脱机输入外围控制机 |
| 输出进程 SPo | 模拟脱机输出外围控制机 |

打印机 SPOOLing：

```text
用户进程输出打印内容
        │
        ▼
快速写入磁盘输出井
        │
        ▼
用户进程无需等待真实打印完成，可继续执行
        │
        ▼
打印机空闲时
        │
        ▼
输出进程 SPo 将输出井内容送入输出缓冲区
        │
        ▼
打印机慢速打印
```

| 特点 | 说明 |
| --- | --- |
| 高速虚拟 I/O | 应用程序快速写入磁盘队列，不必等待慢速设备 |
| 独享设备共享化 | 多进程把作业提交到队列，设备依次处理 |
| 实际 I/O 与程序执行分离 | 虚拟 I/O 时间和真实设备 I/O 时间分离 |

#### 5. 练习题：中断驱动打印机是否合理？

题目条件：每页 50 行、每行 80 字符；打印机每分钟 6 页；写输出寄存器耗时忽略；每打印 1 个字符中断 1 次；单次中断服务耗时 \(50\ \mu\text{s}\)。

第一步：单页字符数：

$$
50 \times 80 = 4000\ \text{个字符}
$$

第二步：每分钟打印字符数：

$$
6 \times 4000 = 24000\ \text{个字符/分钟}
$$

第三步：1 分钟换算为微秒：

$$
1 \times 60 \times 10^6 = 6 \times 10^7\ \mu\text{s}
$$

第四步：单个字符平均打印时间：

$$
\frac{60\times10^6}{24000}=2500\ \mu\text{s}
$$

第五步：中断开销占比：

$$
\frac{50}{2500} \times 100\% = 2\%
$$

结论：中断占用仅为 2%，开销低，不会明显加重 CPU 负担，因此使用中断驱动 I/O 合理。

💡 **解题模板：**

```text
1. 先算每个数据单位的平均产生/处理周期
2. 再算每次中断服务时间
3. 用 中断服务时间 / 数据单位周期 得到 CPU 开销占比
4. 占比较低 → 中断驱动合理
   占比较高 → 中断过频，应考虑缓冲或 DMA
```

### ✅ 本章速记总结

1. I/O 管理的本质是在用户进程和复杂硬件之间建立统一、安全、高效的抽象。
2. 设备控制器通过数据寄存器、控制寄存器、状态寄存器与 CPU 交互。
3. 内存映射 I/O 不能对设备寄存器使用 cache，否则会读到过期状态。
4. 四种 I/O 控制方式体现 CPU 干预程度逐步降低：PIO → 中断 → DMA → 通道。
5. I/O 软件层次：用户进程、设备无关软件、设备驱动、中断处理、硬件。
6. 缓冲技术用于匹配速度、减少中断、提高并行性，但会带来多次复制成本。
7. 单缓冲时间为 $\max(C,T)+M$；双缓冲稳态节拍为 $\max(T,M+C)$。
8. 设备分配按 SDT → DCT → COCT → CHCT 建立完整 I/O 通路。
9. SPOOLing 用软件抽象把独享设备改造成具有共享特征的虚拟设备。
