## 模块一：调度的基本概念与上下文切换 
**1. 调度的时机 (When)**
*   **触发事件**：进程创建/退出、等待I/O、时钟中断（时间片用完）、进程执行中出现异常等【顺便梳理：严谨说法：用户调用、陷阱、中断】。
*   **抢占式 vs 非抢占式**：
    *   **非抢占式**：进程一旦获得CPU，除非自己主动阻塞或完成，否则一直运行。
    *   **抢占式**：系统可强行剥夺正在运行进程的CPU，分配给优先级更高的进程。

**2. 上下文切换 (Context Switch) **
*   **核心步骤**：保存当前进程上下文 $\rightarrow$ 更新PCB并移入合适队列 $\rightarrow$ 选择新进程 $\rightarrow$ 将新进程设为运行态 $\rightarrow$ 恢复新进程上下文。
*   **开销**：
    *   **直接开销**：保存/恢复寄存器、切换地址空间等。
    *   **间接开销 (常考)**：高速缓存(Cache)、缓冲区缓存以及TLB失效导致的性能下降。从一个进程到另一个进程的上下文切换，**必须在内核模式下执行**操作系统代码才能完成。

**3. 调度的性能指标 💡[计算题基础]**
*   **面向用户**：**周转时间**（作业提交到完成的时间）【批处理系统】、**响应时间**（键盘输入到首次输出的时间）【分时系统】、截止时间【实时系统：保证在一定时间内完成】、公平性、优先级。
*   **面向系统**：吞吐量（单位时间完成作业数）【批处理系统】、CPU（处理机）利用率【大中型机器，闲着就是亏】、各种资源的均衡利用（CPU-Bound / I/O-Bound 之间的搭配）【大中型主机】。
*   **带权周转时间**：周转时间 / 实际运行时间。

## 模块二：批处理系统的经典调度算法 
**1. 进程的分类**
A. 资源需求
+ CPU-bound
+ I/O-bound
B. 交互性
+ 批处理：CPU-bound为主
+ 交互式：I/O-bound为主
+ 实时性：如视频/音频
【按照原理、计算模拟甘特图、优劣势分析、适用场景与系统】
**1. 先来先服务 (FCFS)**
类比页面调换的FIFO
*   **特点**：最简单的非抢占式算法，按到达顺序执行。
*   **优点**：简单，性能开销小
*   **痛点**：有利于长作业和CPU繁忙型作业；**极度不利于短作业和I/O繁忙型作业**（会导致I/O设备闲置）。

**2. 最短作业优先 (SJF) & 最短剩余时间优先 (SRTF)**
*   **SJF (非抢占)**：优先调度执行时间最短的进程。**优点** 最小化平均周转时间，提高**吞吐量**；**缺点** 对长作业不利、难以估计作业时间、没有优先级。
*   **SRTF (抢占式)**：SJF的抢占版本。当新到达的进程剩余时间比当前进程还要短时，直接抢占CPU。优点：同上；缺点：源源不断的短作业会导致长作业可能饿死（饥饿）【因为没有优先级机制，HRRF弥补了这一点】

**3. 最高响应比优先 (HRRF) High response ratio First**
*   **核心公式**：响应比 RP = (已等待时间 + 要求运行时间) / 要求运行时间 = **1 + (已等待时间 / 要求运行时间)**。
*   **优势**：FCFS和SJF的折中。短作业与长作业平衡：短作业响应比高；长作业随着等待时间增加，响应比也会提升，**避免了饥饿现象**。
*   **缺点**：性能开销大（每次调度需要重新计RR）

## 模块三：交互式系统的调度算法 
**1. 时间片轮转 (RR - Round Robin)**
*   **特点**：按照FCFS将就绪进程排队，为每个进程分配一个时间片($q$)，时间片耗尽则强制切换并放入队尾。
*   流程：排队、轮转、中断、抢占；出让（主动）
**时间片长度变化的影响**：
- 过长 → 退化为FCFS算法，进程在一个时间片内都执行完，响应时间长。
- 过短 → 用户的一次请求需要多个时间片才能处理完，上下文切换次数增加，响应时间长。
【知道清楚，我们设计的目的】

+ 影响时间片的因素：
  + 预设系统响应时间：$T(响应时间) = N(进程数目) * q(时间片)$
 
    $q(时间片) = \frac{T}{N}$
  + 系统处理能力：CPU执行速度（计组）

**优先级算法**
+ 【根据讲稿与PPT补全这部分笔记】
**优先级分类：动态、静态**

接下来综合优先级与RR算法 我们可以得到MLFQ
**3. 多级反馈队列 (MLFQ - Multi-level Feedback Queue) ⚠️[核心机制]**
MLFQ是现代操作系统的基石。为了在不知道进程执行长度的情况下优化周转时间和响应时间，它设定了极其精妙的规则：
*   **规则1/2**：高优先级队列先执行；同级队列内部采用RR轮转。
*   **规则3 (初始最高)**：新进程进入系统时，直接放入最高优先级队列（假设它是短任务/交互式任务）。
*   **规则4 (降级机制)**：一旦进程用完了该层的**时间时间片**（不论期间主动放弃了多少次CPU），优先级立刻下降一级（防止恶意进程利用I/O操作保留在高优先级【这个似乎不正确，如果在时间片内中断，优先级保持不变，这样也可能造成饥饿】）。
    *   最后一级：不会再降级，时间片耗完重新进入本级的Ready队列。
*   **规则5 (优先级提升/Boost)**：每隔一段时间$S$，将系统中**所有**进程重新移入最高优先级队列（一举解决长作业饿死问题，并适应进程行为的动态变化）。
【级别越低，时间片长度越长，这样可以使得每一个进程能够找到的合适的时间片长度】

**优先级倒置现象**
高优先级进程被低优先级进程延迟阻塞
例子：C上锁，时间片轮转调度但未释放锁，A拿不到锁
+ 解决方法：
  + 优先级倒置
  + 优先级继承

## 模块四：公平共享调度与Linux演进
**1. 公平共享调度 (Fair-share)**
*   **彩票调度 (Lottery)**：给进程发“彩票”，随机抽奖。持有彩票越多，获得CPU概率越高。简单但**短期存在不确定性**。
*   **步幅调度 (Stride)**：确定性的公平算法。计算 `Stride = MaxStride / ticket`，每次运行后累加给 `Pass` 值，调度器总是选择 `Pass` 值最小的进程执行。

**2. Linux 调度的演进 (了解)**
*   **O(n) $\rightarrow$ O(1)**：O(1)利用140个优先级队列（活动/过期数组）和位图，实现了常数级别的极速调度。
*   **CFS (完全公平调度器) 💡[现行Linux标准]**：摒弃传统优先级队列，为每个进程维护一个**虚拟运行时间 (vruntime)**，并使用**红黑树**管理（复杂度为 $O(\log n)$）。调度器永远选择红黑树最左侧（vruntime最小）的进程执行。

## 模块五：实时调度
**实时系统**
+ 计算机必须在**确定的时间范围**内做出反应。
+ 硬实时系统/软实时系统
+ 
**1. 实时调度 (RMS vs EDF)**
**重点：要感知优先级，本质是调整不同任务的offset，由于实时，因此可调度是有条件的**
【用latex改写】
事先确定以下参数，假设一任务集 S = { t₁, t₂, t₃, …, tₙ }，
- 每个任务出现的周期分别是 T₁, T₂, …, Tₙ
- 执行时间为 C₁, C₂, …, Cₙ (Ci < Ti)
- 截至周期(deadline)为 D₁, D₂, …, Dₙ，令 Dᵢ = Tᵢ
- CPU利用率：U = Σ (Cᵢ / Tᵢ)
- 任务独立、上下文切换可忽略

+   **静态表调度**：通过对所有周期性任务的分析预测（到达时间、运行时间、结束时间、任务间的优先关系），事先确定一个固定的**调度方案**（本质上是确认各个任务的offset）。 
    +   无需计算开销小、没有灵活性。

*   **单调速率调度 (RMS) Rate Monotonic Scheduling**：
    *   **静态优先级**
    *   充分条件：Σ (Cᵢ / Tᵢ) ≤ n(ⁿ√2 - 1)
    *   **周期越短，优先级越高**。
    +   **抢占式**
    *   优点：是**单核下静态调度的最优算法**，但CPU利用率有理论上限（约69%）。
    *   适用：硬实时系统（如航天控制），确定性
    *   缺点：有的长周期任务可能错过ddl
  ![18](assets/image-18.png)

*   **最早截止期优先 (EDF) Earliest Deadline First**
    *   条件：有可用的CPU - iff：Σ (Cᵢ / Tᵢ) ≤ 1
    *   **动态优先级**
    *   **绝对截止时间越近，优先级越高**。理论上可以达到100%的CPU利用率。
    *   **抢占式：如果新任务【某一任务下一周期开始】离截止时间更近，则抢占**
    *   缺点：实现相对复杂（优先队列、堆），动态处理；过载行为难预测，低优先级因截止时间早而抢占关键任务
![19](assets/image-19.png)

**2. 优先级倒置 (Priority Inversion) ⚠️[高频名词解释]**
*   **场景**：高优先级任务A想获取资源锁，被低优先级任务C占用（A阻塞）；此时中优先级任务B抢占了C的CPU，导致A被无限期延迟（如著名的NASA火星车重启故障）。
*   **解决方案**：
    *   **优先级置顶 (Priority Ceiling)**：占用资源的进程，优先级自动提至所有可能申请该资源进程的最高优先级。
    *   **优先级继承 (Priority Inheritance)**：低优先级进程C临时“继承”被它阻塞的高优先级进程A的优先级，直到释放资源。

## 模块6 多核调度 
**3. 多处理机调度 (Multi-Processor)**
**注重整体运行效率，多处理机访问数据互斥**
**AMP 非对称式多处理：多个处理器地位不同**
+ 主从处理机（例子：神威太湖之光）
**SMP 对称式多处理系统：~相同**
**自调度：整个系统有一个公共的就绪队列，每个处理机从中选取一个进程运行**
    + 变种：全局就绪队列+ 局部就绪队列（例如每个处理机采用RR），局部就绪队列优先调度
    + 优点：不需要专门分派任务的处理机
    + 缺点：Cache切换开销 队列同步开销

 **Gang scheduling 成组调度**
+ 将**一个进程内的一组线程**，每次分派/剥夺时，都同时分派/从..剥夺 到**一组处理机**上
  + 两种调度：面向进程平分处理机时间，面向线程平分处理机
![20](assets/image-20.png)


**专用处理机调度**
+ 类比表调度，为每个线程固定分配某个处理机。

**单队列 Single queue Multiprocessor Scheduling**【就是上面所说的公共队列

**多队列 MQMS**【每一个处理机有独立的就绪队列】
    
*   **Cache 亲和性 (Cache Affinity)**：尽量让进程保持在同一个CPU上运行，以利用L1/L2 Cache中的遗留状态，避免Cache失效带来的巨大惩罚。
*   **负载平衡（load balance）**：不平衡时，CPU有的负荷大，有的闲置

**平衡负载的方法：**
+   **推：工作迁移 (Work Migration)**: OS 周期性检验不同CPU的负载，不平衡时将超载CPU内就绪队列的一部分进程推到空闲CPU上【系统完成】
*   **拉：工作窃取 (Work Stealing)**：多队列调度(MQMS)中，为了解决某些CPU过载而其他CPU闲置的问题，空闲CPU的队列会去别的繁忙队列中“偷”任务过来执行。
