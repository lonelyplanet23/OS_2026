
# 📚 《操作系统》内存管理 (3) & (4) 

## 模块一：页式存储管理进阶 (Lecture 3核心)

在此之前，我们知道基本分页机制会将逻辑空间分成“页”，物理空间分成“存储块（页框）”。但基本分页面临两个核心痛点：**页表太大（空间开销）** 和 **访存太慢（时间开销）**。

### 1. 页面大小的选取博弈 💡[重点概念]
*   **较小页面**：优点是内存碎片少（仅有极小的页内碎片）；缺点是页表变长，占用内存大，且频繁的磁盘I/O导致换入换出速度慢。
*   **较大页面**：优点是页表短，磁盘I/O换入换出快；缺点是页内碎片大，浪费内存。
*   **最优页面大小公式**：设进程平均占用 $s$ 字节，页表项大小 $e$ 字节，最优页面大小 $p = \sqrt{2es}$。

### 2. 解决空间问题：多级页表 💡[考试计算重点]
*   **痛点**：对于32位系统，若页面4KB，一个进程的页表可能高达4MB，且要求**连续**内存存放。
*   **原理**：将页表本身再进行分页（即“页目录”或外部页表），离散存放在不同物理块中。运行时只需将页目录常驻内存，内部页表按需动态调入。
*   **地址结构**：如32位系统的“10-10-12”结构（一级页表号10位 + 二级页表号10位 + 页内偏移12位）。

### 3. 解决时间问题：快表 (TLB) 与 MMU 💡[高频考点]
*   **痛点**：纯分页机制下，访问一个数据需要 **2次访存**（查页表1次 + 访问真实数据1次）；二级页表需要 **3次访存**，效率严重下降。
*   **机制**：引入MMU（存储管理单元），其内部包含TLB（快表，即页表的高速缓存）。
*   **工作流程**：CPU生成逻辑地址 -> 查TLB -> 若命中直接出物理地址（仅需1次实际访存）；若未命中（TLB miss） -> 查内存中的外部页表并重填入TLB -> 再次映射物理地址。

### 4. 特殊页表结构
*   **哈希页表**：结合哈希函数进行地址转换，解决地址空间过大问题。
*   **反置页表 (Inverted Page Table)**：
    *   **原理**：依据物理内存的**物理页面号**来组织，而不是逻辑页号。
    *   **特点**：表的大小**只与物理内存大小相关**，与逻辑地址空间和进程数无关，极大地节省了页表空间。
    *   **缺点**：查找困难，且**很难实现页面共享**（因为一个物理帧只对应一个虚拟页表项）。

## 模块二：分段存储管理 (Lecture 4核心)

程序中的指令、数据、堆栈对内存管理的需求不同，这就引入了按逻辑意义划分的“分段”机制。

### 1. 分段机制原理
*   **逻辑结构**：作业的地址空间由多个段构成，每个段是一段连续的地址空间，地址结构为二维：**段号 S + 段内位移 W**。
*   **段表**：记录了段与内存物理位置的对应关系。包含：段号、段长（Limit）、段基址（Base）。
*   **地址变换**：
    1. 逻辑地址分离出段号 $S$ 和段内地址 $W$。
    2. 判断 $S$ 是否大于段表长度（越界中断）。
    3. 查段表找到该段基址和段长，判断 $W$ 是否大于段长（段内越界中断）。
    4. 物理地址 = 段基址 + $W$。

### 2. 分页与分段的终极对比 💡[必考核心分析]
| 比较维度 | 页式存储管理 | 段式存储管理 |
| :--- | :--- | :--- |
| **目的** | 实现非连续分配，解决内存碎片问题 | 更好满足用户需要（共享、保护、动态增长） |
| **信息单位** | **页是物理单位** | **段是逻辑单位** |
| **大小** | **固定**（由系统/硬件决定） | **不固定**（由用户程序决定，可动态增长） |
| **地址空间** | **一维**（单一线性空间） | **二维**（段号+段内偏移） |
| **可见性** | 对用户/程序员**透明（不可见）** | 对用户/程序员**可见** |

### 3. 段共享与可重入代码
*   **优势**：分段系统极易实现共享和保护。只需在不同进程的段表中为同一个逻辑程序（如文本编辑器）设置指向同一物理地址的段表项即可。
*   **可重入代码（Pure Code）**：一种允许多个进程同时访问的代码，执行期间**绝对不允许任何进程对其进行修改**（只读）。

## 模块三：段页式存储管理与架构实例 (Lecture 4核心)

### 1. 段页式存储管理
*   **基本思想**：结合二者优点。用**分段方法**分配和管理逻辑虚拟地址，用**分页方法**分配和管理物理实内存。
*   **地址结构**：段号 S + 段内页号 P + 页内地址 W。
*   **数据结构**：每个进程一张段表，**每个段一张页表**。
*   **访存开销**：如果不使用TLB（快表），读取一个字节需要 **3次访存**（查段表 -> 查页表 -> 读数据）。

### 2. X86架构实例 (Intel X86) 💡[拓展与进阶]
*   **宏观过程**：逻辑地址 $\xrightarrow{分段机制}$ 线性地址 $\xrightarrow{分页机制}$ 物理地址。
*   **GDT与LDT**：全局描述符表(GDT)和局部描述符表(LDT)用于段式映射。LDT其实是嵌套在GDT中的一个段。
*   **控制寄存器**：
    *   **CR0**：其中的 `PG` 位用于开启或关闭分页机制。
    *   **CR3**：又称 PDBR (页目录基址寄存器)，存放当前进程**页目录表的物理基地址**，是地址转换的关键起点。
*   **Linux的妥协（扁平化内存）**：为了可移植性，Linux 在X86上让分段机制“形同虚设”（段基址全设为0，段限长为4GB），使得**虚拟地址与线性地址总是一致的**。

## 🚨 期中/期末高频易错考点提炼 (结合历年真题)

1.  **关于 TLB 与 访存次数的计算**：
    *   **考点**：题目常问“成功执行一条Load指令，最少/最多需要几次物理访存”。
    *   **解题法则**：对于二级页表系统，若TLB未命中，需要3次访存（外部页表+内部页表+物理数据）；若TLB命中，则只需 **1次** 访存。如果题目是段页式，未命中则需3次（段表+页表+数据）。
2.  **多级页表逻辑地址映射计算**：
    *   **考点**：给定页目录大小和页大小，求需要多少级页表，或计算特定虚拟地址对应的页目录项/页表项的地址。
    *   **防坑点**：注意题目中页表项（PTE）占用的大小（通常是4字节），计算偏移时是用 `页号索引 * 页表项大小`。
3.  **反置页表(Inverted Page Table)的优劣辨析**：
    *   **易错题**：判断反置页表的优点。
    *   **正解**：优点是**页表与逻辑地址空间大小无关**（只看物理内存大小），缺点是**不便于进程之间共享数据**、查找慢（需要哈希）。
4.  **关于地址空间的判断题**：
    *   **判断题1**：分页存储管理技术的逻辑地址是二维的。 $\rightarrow$ **错误 ❌**。分页地址是一维的，分段才是二维的。
    *   **判断题2**：地址空间不能小于物理内存空间。 $\rightarrow$ **错误 ❌**。地址空间（由指针长度决定）和实际物理内存大小没有必然关系，可以大也可以小。
