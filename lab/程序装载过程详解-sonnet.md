## 一、从源代码到可执行文件的旅程

```
源程序(.c) → [编译器] → 汇编文件(.s) → [汇编器] → 目标文件(.o) → [链接器] → 可执行文件
```

### 1. 编译阶段（Compiler）
- **输入**：C 源代码
- **输出**：汇编代码（.s 文件）
- **工作**：词法分析、语法分析、语义分析、代码优化、代码生成

### 2. 汇编阶段（Assembler）
- **输入**：汇编代码
- **输出**：可重定位目标文件（.o 文件）
- **工作**：将汇编指令翻译成机器码，生成 ELF 格式的目标文件

### 3. 链接阶段（Linker）

#### 静态链接的具体步骤
1. **符号解析**：链接器查找每个符号引用的定义
2. **重定位**：
   - 将所有 .o 文件的同类 section 合并
   - 为每个符号分配最终的虚拟地址
   - 修改所有对符号的引用，使其指向正确的地址

#### 重定位条目
```c
typedef struct {
    Elf32_Addr r_offset;  // 需要修改的位置偏移
    Elf32_Word r_info;    // 符号索引 + 重定位类型
} Elf32_Rel;
```

#### 重定位类型示例（MIPS）
- `R_MIPS_26`：26 位直接跳转地址（jal 指令）
- `R_MIPS_HI16`：高 16 位（lui 指令）
- `R_MIPS_LO16`：低 16 位（addi 等指令）

## 二、ELF 文件格式深入解析

### 1. ELF 文件的两种视图

| 视角 | 组成单位 | 阶段 | 用途 |
|------|----------|------|------|
| **链接视图** | Section（节） | 编译链接 | 链接器处理 |
| **执行视图** | Segment（段） | 加载运行 | 加载器处理 |

### 2. ELF 文件头部结构
```
ELF Header (固定位置)
├── e_ident[16]    // 魔数 + 其他标识
├── e_type          // 文件类型（可执行/可重定位/共享目标）
├── e_machine       // 体系结构（MIPS/x86等）
├── e_version       // 版本
├── e_entry         // 入口点地址（_start）
├── e_phoff         // 程序头表偏移
├── e_shoff         // 节头表偏移
├── e_flags         // 处理器特定标志
├── e_ehsize        // ELF 头大小
├── e_phentsize     // 程序头表项大小
├── e_phnum         // 程序头表项数量
├── e_shentsize     // 节头表项大小
├── e_shnum         // 节头表项数量
└── e_shstrndx      // 节名字符串表索引
```

### 3. 程序头表（Program Header Table）

```c
typedef struct {
    Elf32_Word  p_type;   // 段类型（LOAD = 需要加载）
    Elf32_Off   p_offset; // 文件中的偏移
    Elf32_Addr  p_vaddr;  // 虚拟地址
    Elf32_Addr  p_paddr;  // 物理地址（通常忽略）
    Elf32_Word  p_filesz; // 文件中的大小
    Elf32_Word  p_memsz;  // 内存中的大小
    Elf32_Word  p_flags;  // 权限（R/W/X）
    Elf32_Word  p_align;  // 对齐要求
} Elf32_Phdr;
```

#### 典型的 LOAD 段
1. **代码段**（Text Segment）：
   - `p_flags = R | X`（可读可执行）
   - 包含 `.text` 节

2. **数据段**（Data Segment）：
   - `p_flags = R | W`（可读可写）
   - 包含 `.data` + `.bss` 节
   - `p_memsz > p_filesz`（多出部分是 BSS，初始化为 0）

## 三、程序装载的完整流程（execve 系统调用）

### 步骤 1：Shell 调用 fork() 创建子进程
```
shell (父进程)
    ↓ fork()
    ├─→ shell (父进程，继续运行)
    └─→ shell (子进程，等待 execve)
```

### 步骤 2：子进程调用 execve() 系统调用

#### A. 验证与读取 ELF 文件
1. 检查文件是否存在、是否有执行权限
2. 读取 ELF 头部前几个字节，验证魔数 `0x7f 'E' 'L' 'F'`

#### B. 读取程序头表
1. 根据 `e_phoff` 和 `e_phnum` 读取所有程序头表项
2. 查找类型为 `PT_LOAD` 的段

#### C. 创建虚拟地址空间
1. 销毁原有的虚拟地址空间
2. 为新程序建立新的页表
3. 此时还没有分配物理页框，只是建立了映射关系

#### D. 映射 LOAD 段到虚拟地址空间
对每个 `PT_LOAD` 段：
1. 计算虚拟地址范围：`[p_vaddr, p_vaddr + p_memsz)`
2. 根据 `p_flags` 设置页表项权限（R/W/X）
3. 此时并不从文件读取内容，只是建立映射！（按需调页的关键）

#### E. 处理 BSS 段
- 当 `p_memsz > p_filesz` 时，多出的部分就是 BSS
- 这部分虚拟地址空间被标记为 "匿名页"（Anonymous Page）
- 第一次访问时会分配物理页并清零

#### F. 设置堆栈
1. **用户栈**：
   - 分配虚拟地址空间（通常在高地址）
   - 大小由系统限制（如 8MB）
   - 将 `argc`、`argv`、`envp` 压入栈中

2. **内核栈**：为进程分配内核栈（用于系统调用）

#### G. 准备进程控制块（PCB）
1. 设置程序计数器（PC）为 `e_entry`（通常是 `_start`）
2. 设置栈指针（SP）指向用户栈顶
3. 初始化其他寄存器

#### H. 返回到用户态
- CPU 切换到用户态
- 从 `_start` 开始执行

## 四、程序启动：从 _start 到 main

### 1. _start 函数（链接器提供）

**_start 的主要工作**（以 MIPS 为例）：
```assembly
_start:
    # 1. 从栈中获取 argc, argv, envp
    lw   $a0, 0($sp)   # argc
    addi $a1, $sp, 4   # argv
    addi $a2, $a1, 4   # envp

    # 2. 初始化 C 运行库（libc）
    jal  __libc_init_first
    jal  __cxa_atexit_init

    # 3. 调用 main 函数
    jal  main

    # 4. main 返回后，调用 exit()
    move $a0, $v0       # main 的返回值作为 exit 参数
    jal  exit            # exit() 不会返回
```

### 2. main 函数执行
- 这是程序员编写的代码
- 参数 `argc`、`argv` 由 `_start` 传递

### 3. 退出流程
- `main` 返回 → `exit()` → 内核回收资源

## 五、动态链接的装载过程（进阶）

### 1. 动态链接器本身也是一个共享库
- 路径通常是 `/lib/ld-linux.so.2`（32位）或 `/lib64/ld-linux-x86-64.so.2`（64位）
- 它也是一个 ELF 文件，但比较特殊（可以自举）

### 2. 动态链接的装载步骤

当可执行文件是动态链接时（`.interp` 段存在）：

#### A. 内核加载动态链接器
1. 读取可执行文件的 `.interp` 段，找到动态链接器路径
2. 加载动态链接器到虚拟地址空间
3. 将控制权交给动态链接器的 `_start`

#### B. 动态链接器进行重定位
1. 对自身进行重定位（因为它也是共享库）
2. 加载程序依赖的共享库（如 libc.so）
3. 对所有共享库进行重定位
4. 解析符号引用（如 `printf` 的地址）

#### C. 跳转到程序的 _start
- 动态链接器完成工作后，跳转到可执行文件的 `_start`

### 3. PLT 和 GOT（延迟绑定）

#### 第一次调用函数时
```
调用 printf → PLT[0] → 查找并填充 GOT[0] → 调用真实 printf
```

#### 后续调用
```
调用 printf → PLT[0] → 直接跳转到 GOT[0] 中的真实地址
```

## 六、缺页异常与按需调页

### 1. 为什么需要按需调页？
- 程序通常不会同时使用所有代码和数据
- 节省物理内存
- 加快程序启动速度

### 2. 缺页异常处理流程

```
1. CPU 访问虚拟地址
   ↓
2. MMU 查页表 → 发现页不存在（Present = 0）
   ↓
3. CPU 触发**缺页异常**（Page Fault）
   ↓
4. 进入内核态，调用缺页异常处理程序
   ↓
5. 处理程序判断：
   ├─ 合法访问 → 继续
   └─ 非法访问 → Segmentation Fault (SIGSEGV)
   ↓
6. 分配物理页框
   ↓
7. 从文件读取内容到物理页（如果是文件映射）
   或 清零（如果是匿名页，如 BSS/堆/栈）
   ↓
8. 更新页表项，设置 Present = 1
   ↓
9. 返回用户态，重新执行那条指令
```

## 七、完整的内存布局示例

```
高地址
    ┌─────────────────┐
    │  环境变量表     │ envp
    ├─────────────────┤
    │  命令行参数     │ argv
    ├─────────────────┤
    │  栈 (Stack)     │ ← 向下生长
    │                 │
    │  ... (空洞) ... │
    │                 │
    │  堆 (Heap)      │ ↑ 向上生长
    ├─────────────────┤
    │  .bss 段        │ 未初始化全局变量（清零）
    ├─────────────────┤
    │  .data 段       │ 已初始化全局变量
    ├─────────────────┤
    │  .text 段       │ 代码（只读、可执行）
    ├─────────────────┤
    │  保留区域       │
低地址
```

## 总结

程序装载的核心步骤：
1. **编译链接**：生成 ELF 可执行文件
2. **fork**：创建子进程
3. **execve**：
   - 读取 ELF 头部
   - 建立虚拟地址空间映射（LOAD 段）
   - 设置堆栈
   - 准备寄存器
4. **动态链接**（如果需要）：加载共享库、重定位
5. **_start**：初始化 libc，调用 main
6. **按需调页**：访问时才真正分配物理内存
